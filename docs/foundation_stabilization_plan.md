# FluentFlow 地基稳定化计划

本文件是"如何止血、如何把地基收敛牢固"的执行层计划，是多会话协作时的共享事实源。
它与 `docs/current_version_plan.md`（产品阶段与边界）、`docs/task_list_reconciliation_plan.md`
（任务列表单一事实源重构）对齐，不另起炉灶：current_version_plan 回答"做什么"，本计划回答
"先别做什么、把什么修牢、并行时谁改哪块"。

飞书镜像（供人阅读/掌舵）：
`https://skardilabs.feishu.cn/wiki/FkUlwgl1ui8fSDkI9necW0P8nlg`。两处内容应保持一致；仓库版是
执行会话实际读取的权威副本。

## 一、这份计划解决什么

FluentFlow 从"本地视频转写 → 笔记 → 导入飞书"这一条清晰链路起步，后来快速叠加了云端转写、
视频链接抓取、Agent 工作流、多账号、云工作区代理等能力。功能长得比地基快，导致现在不断修 bug、
修完又换个地方冒出来。

目标不是"把 bug 修得更快"，而是把反复出问题的少数地基收敛成稳定的几根脊柱，并用测试焊死，让
同一类问题不再复发。地基稳住之前，不加新功能。

## 二、核心判断：bug 是少数地基的重复显形

2026-07-08 一天里出现的问题——多视频排队失败、删除记录复活、云端转写提交被拒（401）、上传卡死、
云代理挂起——表面是 5 个独立 bug，根子其实只有 3 类结构性弱点：

- **任务状态与缓存没有单一事实源**：前端多层缓存（history / account cache / warm cache）与后端
  列表取并集合并，导致已删除的记录被重新并回来。
- **认证与执行上下文的判断散落、且与"本地/云转写"耦合**：本地执行豁免依赖一个只在本地转写时才
  发送的请求头，逻辑散落在白名单、中间件和两份重复的 `localExecutionHeaders` 里，切到云转写就 401。
- **处理流水线曾靠 HTTP 回调自身执行**：队列 worker 通过给自己发 HTTP 请求来跑任务，引入了认证
  边界、令牌绕过、自调用这一整类脆弱面。

结论：不是 bug 多，是同几个弱点在多处显形。只填症状、不动根，就会换个地方再冒。

## 三、四根脊柱与收敛目标

把整个系统的可靠性归结到 4 根脊柱。每根都必须收敛成"唯一权威实现"，其它地方只引用、不各自造一套。
4 根里已有 3 根在最近开始收敛，本计划是要把它们有意识地做完。

| 脊柱 | 现状 | 目标 | 验收标准 |
| --- | --- | --- | --- |
| 任务状态 | 单一事实源重构进行中（AppProvider 已成为任务列表唯一拥有者） | 后端 job store 为唯一权威，前端只投影 | 增删改刷新后与后端一致，有测试 |
| 执行/认证上下文 | 最散，且与转写方式耦合 | 收敛为一处"是否本地执行"判断，与 STT provider 解耦 | 本地下本地/云转写都能提交，公网仍需登录 |
| 处理流水线 | 已从 HTTP 自调用抽成进程内 runner | worker 与 `/process` 共用同一流水线，无自调用 | 队列串行、无内部 HTTP 自调用，有端到端验证 |
| 数据流/缓存 | reconcile 重构进行中（tombstone 剪枝） | 一条缓存写路径 + tombstone | 删除后刷新不复活，有 vitest 覆盖 |

主战场文件（用于划分 owner，避免并行踩踏）：

- 任务状态 + 数据流/缓存：`frontend/src/app/AppProvider.jsx`、`frontend/src/lib/jobMappers.js`、
  `frontend/src/app/jobMorph.js`、后端 `backend/core/job_store.py`。这两根实为一体，不拆成两条并行流。
- 执行/认证上下文：`backend/core/request_scope.py`、`backend/core/server_helpers.py`（中间件）、
  `frontend/src/lib/localExecution.js` 与 `frontend/src/app/shared.jsx`（两份 header，需合并为一处）。
- 处理流水线：`backend/routers/processing.py`、`backend/core/server_helpers.py`（worker）。

## 四、执行纪律（五条）

- **1. 冻结新功能**：给自己划一段时间（建议 2～3 周），不加任何新能力。地基漏水时继续盖楼只会更快沉。
- **2. 收敛脊柱为唯一实现**：上表 4 根，每根做成一处权威实现，删掉重复副本（例如两份
  `localExecutionHeaders`）。
- **3. 每修一个 bug 补一道守卫**（Harness First）：同一个 bug 不允许被修第二次而没有测试挡着。
- **4. 砍非核心功能**：用第一性原理逐个问"它服务核心 job 吗，还是技术上能做就做了"。每砍一个，
  就少一整类未来的 bug。
- **5. 定义"稳定"的验收线**：写死一句可检验的标准，达标前不解冻新功能（见第七节）。

## 五、功能取舍建议

判断基准：是否服务"本地/云 音视频 → 转写 → 结构化笔记 → 导出"这一条核心 job。以下为 2026-07-08 已拍板的决定。

| 功能 | 决定 | 理由 |
| --- | --- | --- |
| 音视频 → 转写 → 笔记 → 飞书 | 保留（核心） | 产品真正的价值主线 |
| 云端转写（ElevenLabs） | 保留 | 本地转写的稳定补充 |
| 字幕/文本导入 | 保留 | 与核心链路同源，成本低 |
| 视频链接抓取 | 只保稳定平台，其余降级 | 平台多且易碎；不支持的明确拒绝并提示，不假装支持 |
| 云工作区代理 | 保留但默认硬禁用 | 代码留作未来多设备同步；但不从残留环境变量静默启用，开启需显式开关且启动时明确提示 |
| Agent 工作流 / 多账号等 | 暂缓评估 | 非核心，稳定后再决定投不投；稳定期不动 |

核心链路验收（一句话，达标才算稳定）：一段本地转写视频 + 一段云端转写视频，各自跑通导入 → 转写 →
笔记 → 导出，结果能在编辑器打开且内容完整，各连续 3 次不出错。

## 六、分阶段推进

### 阶段 0：冻结与盘点（约 2～3 天）

- 宣布功能冻结，只允许修 bug 和收敛脊柱。
- 把"核心链路"写成一句话验收：导入 → 转写 → 笔记 → 导出，本地和云两种模式各连续跑 N 次不出错。
- 确认第五节的砍/留清单。

### 阶段 1：收敛脊柱（约 2～3 周）

- 逐根脊柱做到"唯一权威实现 + 测试覆盖"，优先级：任务状态 → 执行/认证上下文 → 数据流/缓存
  （流水线已基本完成）。
- 每个已知 bug 修复时补回归测试，删除重复副本和死代码。
- 砍掉第五节里确定要砍的功能。

### 阶段 2：解冻与验收

- 跑第七节核对表，全绿才算稳定。
- 稳定后再逐个评估"暂缓"的功能，每次只加一个，且遵守"每加一个功能必须先想清它落在哪根脊柱上"。

## 七、解冻前必须全绿（核对表）

- [ ] 核心链路（本地转写）：导入 → 转写 → 笔记 → 导出，连续跑通不出错。
- [ ] 核心链路（云端转写）：同上。
- [ ] 多文件排队：串行处理、一次一个，全部完成。
- [ ] 删除处理记录：刷新后不再复活。
- [ ] 上传失败/卡住：有明确提示，不无限转圈。
- [ ] 后端可从干净环境启动（不依赖残留环境变量，如 `FLUENTFLOW_CLOUD_WORKSPACE_URL`）。
- [ ] 前端 vitest + 后端 pytest 全绿。
- [ ] 四根脊柱各自都有至少一条守卫测试。

## 八、并行推进与分工

多个对话/会话可以并行，但 AI 会话之间不共享上下文，因此必须按文件边界切分并以本文件为共享看板。

规则：

- **按文件边界并行，不按"感觉"并行。** 两个会话同时改同一个热点文件必然互相覆盖。
- **`backend/core/server_helpers.py` 同一时间只能有一个 owner。** 它是所有人都要碰的咽喉。
- **一分支一会话**：每个并行会话在自己的分支/worktree 上做，合并时人为审一次。
- **在干净点提交**：并行流建立在"已提交的稳定态"上，不要建立在别人未提交的半成品上。
- **本文件当看板**：开工先读、做完更新第三节"现状"列，避免重复劳动或矛盾决定。
- **留一个集成会话**：所有分支合回后，由一个会话统一跑全量测试 + 过第七节核对表。并行最大的坑
  不是各自没做完，而是各自都以为做完了、合起来却不自洽。

可并行度：

- **完全可并行**（不碰共享文件）：笔记质量评测脚本、文档、孤立的功能删除。
- **耦合的脊柱重构**：最多 1～2 条流，且严格按第三节的主战场文件划 owner。
