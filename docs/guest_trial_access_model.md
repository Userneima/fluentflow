# FluentFlow Guest Trial Access Model

## Context

FluentFlow now has a public domain: `https://fluentflow.icu`.

The product has two simultaneous jobs:

- It is a portfolio project linked from Wang Yuchao's resume and portfolio.
- It is also a real video-to-transcript/note tool with non-trivial compute, storage, and API cost.

Because of the portfolio role, first-time visitors should not be blocked by a login wall. They need to enter the product and experience a real working flow. Because the product calls real STT and LLM services, open access still needs strict capacity and abuse boundaries.

## Product Direction

Use a real guest trial rather than a fake demo.

Guest visitors should be able to:

- Open the product without logging in.
- Submit a small real video/audio file.
- Receive a real transcription and note result generated by the production pipeline.
- Understand what the product does before creating or requesting an account.

The guest experience should feel intentionally scoped, not broken or artificially crippled.

## Scope For This Phase

This document only covers guest trial behavior.

Out of scope for this phase:

- Normal logged-in user quotas.
- Admin quotas and admin override behavior.
- Payment, subscription plans, or public self-serve onboarding.
- Full multi-tenant SaaS architecture.
- Long-term user history and account-based workspace design.

## Current Concern

The current server is a small single ECS instance. The main risks are not normal page views, but:

- Large media uploads.
- FFmpeg extraction.
- Long-running STT tasks.
- Azure Batch transcription cost.
- LLM summary cost.
- Local disk usage for source files and artifacts.
- Queue buildup after public promotion.

The access model should protect the system while still allowing a high-quality portfolio trial.

## Guest Trial Principle

Guests should confirm a queue position first, then upload the file, then wait for transcription and note generation.

The queue should be a controlled trial-processing queue:

1. Visitor opens FluentFlow.
2. Visitor starts a guest trial.
3. Backend confirms that the guest queue has capacity.
4. Visitor uploads one allowed file.
5. The uploaded task enters the guest processing queue.
6. If waiting, the visitor sees queue position and estimated wait.
7. The system starts real processing when previous guest tasks finish.
8. The slot is released when the job completes, fails, is cancelled, or times out.

This avoids the deadlock where the first visitor in line never uploads and blocks everyone behind them. It does mean queued guest files may occupy disk before processing starts, so guest limits, queue length, file size, duration, retention, and cleanup are part of the capacity boundary.

## Recommended Guest Limits

Initial conservative defaults:

- One file per guest trial.
- No batch upload for guests.
- Do not actively encourage consecutive guest trials after completion.
- Real STT and LLM processing.
- File size limit: 150 MB.
- Media duration limit: 15 minutes.
- Active guest processing slots: 1.
- Waiting guest queue limit: 5.
- Per-IP daily guest trial limit: 2.
- Guest result retention: 24 hours.
- Guest result downloads are allowed.
- No Feishu export in guest mode.
- No long-term history in guest mode.

These numbers should be treated as launch defaults, not permanent product strategy.

## Queue Behavior

Guest queue states:

- `ready_to_upload`: the queue has capacity and the visitor may upload one file.
- `uploading`: the visitor is uploading an allowed file.
- `waiting`: the uploaded file is accepted and waiting behind earlier guest tasks.
- `processing`: the visitor has uploaded and a real job is running.
- `completed`: result is ready.
- `failed`: processing failed with a user-readable reason.
- `cancelled`: visitor cancelled the queue or task.
- `expired`: visitor stopped sending heartbeat or the queued upload exceeded retention rules.
- `queue_full`: too many guests are already waiting.

Queue behavior rules:

- Refreshing the page should not lose the visitor's queue position.
- Closing the page briefly and reopening should recover the same queue position if the queue token is still valid.
- Long absence should remove the visitor from the queue.
- A visitor should only upload after the backend confirms queue capacity.
- Once upload completes, the queue position belongs to that uploaded task.
- If the visitor leaves after upload, the task may continue unless explicitly cancelled or expired by backend policy.

Suggested timing:

- Frontend heartbeat interval: 15 seconds.
- Stale queue timeout without heartbeat: 60-90 seconds.
- Upload start window after queue capacity is confirmed: 5 minutes.
- Uploaded waiting-task retention should be short enough to prevent disk buildup if the visitor disappears.

## Queue Identity

Guest queue identity should use both:

- A backend queue token.
- Browser `localStorage` for recovery across refreshes.

Expected recovery flow:

1. Browser stores `guest_trial_token`.
2. On page load, frontend calls a guest trial status endpoint.
3. Backend checks whether the token is still valid.
4. If valid, restore the visitor's queue or processing state.
5. If expired or invalid, create a new guest queue session when the visitor asks to try.

Do not rely only on frontend unload events. Browsers cannot reliably notify the server when a tab closes, so backend heartbeat expiry must be authoritative.

## Wait Time Display

The interface should show waiting users:

- Number of people ahead.
- Estimated wait range.
- Current state.
- Option to cancel.

Use ranges rather than precise promises.

Example:

```text
前面还有 3 个试用任务
预计等待约 24-36 分钟
请保持页面打开；刷新页面不会丢失位置
```

Initial estimate can be simple:

```text
estimated_wait = people_ahead * 8-12 minutes
```

This is acceptable because guest files are capped to short durations. Later, replace it with a rolling average from recent completed guest jobs.

## Guest UX Copy

The guest flow should lead with a positive trial promise:

```text
访客试用
支持 15 分钟以内的视频，真实生成字幕与笔记。
```

If upload exceeds limits:

```text
这个文件超过访客试用范围。
访客试用支持 150MB / 15 分钟以内的单个文件。
```

If the queue is waiting:

```text
前面还有 3 个试用任务。
预计等待约 24-36 分钟。
你的文件已进入试用队列，转录开始后会自动更新进度。
```

If queue is full:

```text
当前试用人数较多。
为了保证生成质量，访客队列暂时已满。请稍后再试。
```

Avoid framing the first screen as "not logged in, cannot use". The product should feel available, with clear trial boundaries.

## Default Guest Prompt

The recommended default prompt for the guest trial and new visitors should be:

```text
当前是一份语音转字幕文件，请你根据这个文件类型生成合适的提示词，并根据这个提示词产出对应的笔记。将最终的笔记内容输出给我即可，无需任何其他内容。
```

This prompt should be available as a built-in prompt option and selected before the older course-note, meeting, research, and quick-bullet templates for new visitors. The older templates may remain available as alternatives, but the default guest experience should avoid making the visitor choose a specialized note type before they understand the product.

## Guest Result Access

Guest trials should complete with a real usable result, not just an on-screen preview.

Guests may download the artifacts generated by their own trial, such as:

- Transcript text.
- Subtitle files.
- Generated Markdown notes.

Guest access should still be temporary:

- The current browser can recover the result through the guest trial token while it remains valid.
- Refreshing the page should restore the completed result if the token is still valid.
- Closing the tab and returning later in the same browser should restore the completed result within the retention window.
- Changing browser or device should not restore the guest result.
- Result links should not become permanent public URLs.
- Guest artifacts should be cleaned up after the guest retention window.
- Guest trials should not appear as long-term account history.

Default retention recommendation:

- Keep guest result access for 24 hours.
- After expiration, remove the guest queue/session record and clean generated artifacts according to the storage cleanup policy.

This makes the trial feel like the real product while avoiding an implicit anonymous storage product.

After a guest trial completes, the interface should prioritize:

- Viewing the generated result.
- Downloading artifacts.
- Explaining how to access the full experience if the visitor wants to process more material.

It should not foreground a "try another file" loop. The backend may still enforce a small daily allowance such as 2 trials per IP, but the product experience should treat one successful trial as the intended portfolio evaluation path.

## Architecture Implication

The current backend already has a processing queue for uploaded jobs. Guest trial should add guest-specific admission control before upload and guest-specific queue visibility after upload.

Conceptual separation:

- Guest admission control: checks whether a visitor may start uploading.
- Guest processing queue: handles accepted uploaded guest jobs.

The guest processing queue may persist uploaded source files while waiting, so file size, duration, queue length, and cleanup must be conservative.

## Current Recommendation

Build the guest trial as the next access-model change:

1. Remove the login wall for first-time visitors.
2. Keep real processing for guest trials.
3. Add guest upload admission, uploaded-task queue visibility, heartbeat, and expiry.
4. Enforce strict guest limits.
5. Keep account/admin behavior out of this phase.

This gives portfolio visitors a real product experience while keeping capacity and cost bounded.

## Implementation Phases

Implement this in stages. Do not combine guest trial, normal logged-in user quotas, admin behavior, and payment into one large change.

### Phase 1: Backend Guest Trial Control

Goal: create the backend safety boundary before changing the public UI.

Build:

- Guest trial admission check before upload.
- Guest queue/session token.
- Guest status endpoint.
- Guest heartbeat endpoint.
- Guest cancel endpoint.
- Guest queue position and estimated wait metadata after upload.
- Guest-specific file size, duration, daily IP, active, and waiting-queue limits.
- Expiry cleanup for abandoned or stale guest records.

Do not build:

- Payment.
- Logged-in user plans.
- Admin dashboard.
- Long-term anonymous history.

### Phase 2: Frontend Guest Entry

Goal: let first-time visitors enter the real product without the login wall.

Build:

- Allow unauthenticated visitors into the main product shell.
- Show guest trial state in the upload/workbench area.
- Keep the primary call to action focused on starting a real guest trial.
- Hide or de-emphasize features that are not part of the guest trial, such as batch upload, Feishu export, local model settings, API key inputs, and long-term history.

Do not turn the first screen into a marketing landing page or a standalone queue page.

### Phase 3: Guest Upload And Queue Experience

Goal: make waiting understandable rather than mysterious.

Build:

- Confirm queue capacity before allowing upload.
- Upload one allowed file.
- Put the uploaded task into the guest processing queue.
- Show number of tasks ahead.
- Show estimated wait range.
- Show upload, waiting, processing, failed, cancelled, expired, and completed states.
- Let the visitor cancel when waiting or processing if backend cancellation is safe.

Use ranges for wait time, not exact promises.

### Phase 4: Guest Result Recovery And Downloads

Goal: complete the portfolio trial loop.

Build:

- Same-browser result recovery through `guest_trial_token`.
- Refresh recovery after completion.
- Return-later recovery within the retention window.
- Download buttons for transcript text, subtitle files, and Markdown notes.
- Expiration behavior after 24 hours.

Do not create permanent public result URLs.

### Phase 5: Validation And Release Guardrails

Goal: verify the public trial cannot easily overload the single server.

Test:

- Fresh unauthenticated visitor can enter the product.
- Guest can start a trial, upload one valid small file, wait, process, complete, and download artifacts.
- Upload over file-size limit is blocked before expensive processing.
- Media over duration limit is blocked with a clear message.
- Batch upload is unavailable for guest.
- Queue-full state appears when waiting limit is reached.
- Refresh preserves waiting or completed state.
- Closing and reopening in the same browser preserves valid state.
- Expired token no longer restores old state.
- Guest result disappears after retention cleanup.
- Existing logged-in/admin paths are not accidentally changed in this phase.

## Launch Defaults

Use these values for the first public portfolio trial release unless there is a concrete reason to change them:

```text
guest_file_limit_mb = 150
guest_duration_limit_minutes = 15
guest_active_processing_slots = 1
guest_waiting_queue_limit = 5
guest_daily_trials_per_ip = 2
guest_result_retention_hours = 24
guest_heartbeat_interval_seconds = 15
guest_stale_timeout_seconds = 90
guest_wait_estimate_per_task_minutes = 8-12
```

These are product defaults, not hard-coded permanent strategy. After the first real usage wave, adjust them using observed queue length, task duration, failure rate, and API cost.
