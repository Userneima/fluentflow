# FluentFlow 产品成长记录

本文档记录 FluentFlow 从“可用工具”走向“可验证产品”的关键迭代。它不是功能清单，而是记录每次改动背后的问题判断、取舍和可沉淀的产品能力。

## 2026-06-02｜Azure 云转录接入

### 问题

本地 STT 速度强依赖设备性能。对长音频来说，同一条产品链路在不同电脑上的等待体验差异很大，简历或报告中的 STT 速度数据也必须说明环境口径。

### 判断

云转录不应该替代本地默认链路，而应该作为可切换的 STT Provider。这样用户可以在“本地隐私/无需云服务”和“云端速度/跨设备稳定性”之间选择，也能让事件日志按 provider 分组比较。

### 方案

- 新增 Azure Speech Fast Transcription 适配层。
- 工作台增加转录服务选择：本地 faster-whisper / Azure 云转录。
- Azure 地址和 Key 只保存到本机后端配置，不写入浏览器 localStorage；用户可填地区名或完整地址，后端统一补全为调用链接。
- Azure 返回结果转换成现有转录结构，继续复用转录清洗、编辑器、摘要和飞书导出。
- Azure 路径开启说话人区分时，使用 Azure Fast Transcription 的 diarization 返回 speaker 标签；本地路径继续使用 pyannote。
- Azure 路径下的“自动识别”不再硬传候选 locale，而是省略 `locales` 交给 Fast Transcription 自动识别主语言；显式 locale 被 Azure 拒绝时，后端会降级为不指定 locale 重试。
- Azure Fast 默认使用 REST API `2025-10-15`；如果当前版本触发 `InvalidModel`，后端会尝试兼容版本 `2024-11-15`。若仍失败，应切回本地 STT 或更换 Azure Speech 资源/区域。
- Azure Fast 路径改用压缩 MP3 inline 上传，并在上传前记录预处理后大小/时长，避免本地 Whisper 专用 WAV 膨胀导致云端上传不稳定。
- 增加 `scripts/check_azure_fast_transcription.py`，用于真实凭证下单文件 smoke test；验证结果不写入事件表，避免把调试样本混入产品使用数据。
- `stt_completed` 增加 `stt_provider` 口径，后续报告必须按本地和 Azure 分组。

### 取舍

Azure Fast Transcription 不提供与 faster-whisper segment 生成同等粒度的真实进度，因此前端只展示云端转录状态和等待时间，不伪造百分比。外部 HTTP 请求也无法像本地 STT 子进程一样被强制杀掉。当前阶段只做 inline 上传，不提前引入 Azure Blob/SAS 配置；超长音频需要未来单独做 Batch 流程。

### 可沉淀表达

为 FluentFlow 增加可切换 STT Provider，将本地 faster-whisper 与 Azure 云转录纳入同一处理链路和数据口径，支持按设备/云端服务对比转录耗时与稳定性。

## 2026-06-03｜Azure 云转录从 Fast inline 转向 Batch + Blob/SAS

### 问题

Azure Fast inline 链路在真实文件体量下暴露出多类不稳定问题：HTTPS 上传可能 EOF/Broken pipe，部分请求返回 `InvalidModel`，并且 Fast 不支持当前产品需要的说话人区分能力。继续在 Fast 上叠加重试和兜底，只会让用户看到“云转录可用但经常失败”的假象。

### 判断

云端长音频转录应该走异步 Batch，而不是同步 inline。Batch + Blob/SAS 的配置成本更高，但它更符合产品真实场景：长录音、大文件、可等待、可轮询、可保留云端任务状态，并且官方 API 支持 diarization 配置。

### 方案

- 新增 Azure Batch + Blob/SAS 转录路径，作为工作台里的正式 Azure 云转录选项。
- 新增 Azure Speech 基础自检入口，先用短音频 REST 识别验证 Speech 资源、区域和 Key，不依赖 Blob/SAS。
- 音视频先压缩成 MP3，再通过 Blob container SAS 上传，随后调用 Speech Batch `transcriptions:submit`。
- 工作台新增 Blob 容器 SAS URL 保存入口；Speech 地址、Key、Blob SAS 都只写入本机后端配置，不写入浏览器 localStorage。
- Dashboard 与编辑器重新转录会在上传前检查 Batch 必需配置，缺失时直接提示，不再让用户等到后端报错。
- Batch 转录阶段拆分为上传 Blob、提交任务、等待 Azure、下载结果；不伪造云端 STT 百分比。
- Azure Batch 可选请求说话人区分；本地路径仍使用 pyannote，Azure Fast 旧链路禁用说话人区分。
- 新增 `scripts/check_azure_batch_transcription.py`，支持 dry-run 检查配置和音频预处理，也支持真实 Batch smoke test。

### 取舍

Batch 真实测试需要 Blob SAS，且会产生 Azure Storage 与 Speech 调用痕迹；因此自动化测试先锁住请求形状和结果解析，真实 smoke test 作为配置齐全后的手动验证步骤。当前本地配置已具备 Speech 地址和 Key，并且基础短音频自检已在 eastasia 资源上通过；但尚未配置 Blob container SAS，无法完成真实 Batch 文件转录提交。

### 可沉淀表达

将 FluentFlow 的云端 STT 从 Fast inline 试验路径迁移为 Azure Batch + Blob/SAS 异步路径，补齐长音频、云端轮询、SAS 配置、说话人区分和真实 smoke test 能力。

## 2026-06-01｜历史快照与数据口径

### 问题

简历中只有“做了一个 AI 工具”的功能描述，缺少能证明数据能力和产品验证能力的真实证据。

### 判断

不能把历史 localStorage 或日志伪造成事件日志。历史数据只能作为快照口径，未来真实使用才进入事件表。

### 方案

- 新增历史任务快照报告能力，从前端 localStorage 和后端日志回溯已有任务规模。
- 指标按 `resume_ready`、`needs_context`、`internal_only`、`unavailable` 分级。
- 报告中显式标注 `source`、`confidence`、`resume_usable` 和口径说明。

### 取舍

优先保证数据可信度，而不是为了简历好看补复杂体系。

### 可沉淀表达

基于历史任务快照和日志口径回溯使用规模，区分高可信、需解释和内部参考指标，避免将历史快照伪造成事件数据。

## 2026-06-01｜第一阶段事件日志

### 问题

需要自然积累真实使用数据，用于验证流程效率、稳定性和导出链路。

### 判断

埋点必须对应真实产品入口，不能假设用户有“人工校对完成、评分、结果可用”等当前不存在的行为。

### 方案

- 新增 best-effort SQLite 事件日志：`data/fluentflow_events.sqlite`。
- 只记录真实路径事件：导入、音频提取、STT 完成、转写就绪、摘要完成/失败/跳过/fallback、飞书导出、下载、取消、任务终态。
- 不保存完整转写文本或完整 Markdown，只保存长度、耗时、状态、错误原因和链接。
- 新增事件导出脚本，支持 JSON / CSV。

### 取舍

第一阶段只验证系统链路，不做质量反馈漏斗。

### 可沉淀表达

设计并落地本地任务事件日志，记录端到端处理阶段、耗时、失败原因和导出结果，为后续效率与稳定性分析提供数据基础。

## 2026-06-01｜设备相关的 STT 性能口径

### 问题

本地转写耗时强依赖设备性能。如果直接写“平均转写耗时”，容易把当前设备表现误当作产品普遍表现。

### 判断

产品报告应该说明“在当前运行环境下”的数据，并记录设备、模型、速度档、冷/热启动等上下文。

### 方案

- `stt_completed` 记录运行环境、模型配置、`stt_realtime_factor`、模型加载耗时和缓存状态。
- 新增 STT 性能报告脚本，支持按设备、模型、速度档和语言分组。
- 新增 benchmark 脚本，用相同核心链路跑单文件基准。

### 取舍

不把设备差异藏到简历里，而是在报告中建立可解释口径。

### 可沉淀表达

为本地 AI 转写链路补充设备与模型维度性能口径，用实时倍率和环境 metadata 解释不同硬件下的体验差异。

## 2026-06-01｜STT 进度反馈

### 问题

用户上传音频后，STT 阶段长时间停在 22%，无法判断是卡死、模型慢，还是等待第一段结果。

### 判断

faster-whisper 在第一段 segment 产出前没有可用的真实音频进度。此时不能伪造百分比，但可以展示内部状态和等待时长。

### 方案

- STT 阶段 SSE 增加 `stt_status` 和 `stt_elapsed_seconds`。
- 前端展示加载模型、准备音频、等待第一段、接收片段等状态。
- 第一段长时间未产出时给出解释性提示。

### 取舍

不伪造进度条，让界面诚实表达“没有 segment 前只能等待”。

### 可沉淀表达

优化长任务可观测性，将黑盒转写阶段拆成可理解状态，降低用户对任务卡死的误判。

## 2026-06-01｜STT 质量回退修正

### 问题

为了解决转录阶段长时间停在 22% 的体验问题，曾尝试将音频默认切成 60 秒小块分别转录。这个方案让进度更容易推进，但实际转录质量反而下降。

### 判断

切块转录改变了 faster-whisper 的输入条件。长音频里的上下文、句子延续和术语判断会被 60 秒边界打断，尤其中文讲座、访谈和经验分享这类连续语境材料，质量损失比进度收益更严重。

### 方案

- 默认 STT 路径切回整段音频转录，恢复跨段上下文。
- 继续保留 STT 子进程，确保用户取消时仍能真正终止推理任务。
- 进度反馈改回质量优先口径：第一段 segment 产出前展示状态和等待时间；产出后按真实 segment 时间戳推进。
- 分块转录能力保留在底层，但不作为默认主流程。
- 热词库默认改为不注入，只有用户明确选择场景热词时才影响 STT。

### 取舍

放弃“每 60 秒必然推进一次”的进度细腻度，换回更稳定的转录质量。对当前产品来说，转录质量比进度条连续移动更重要。热词库也从默认干预改为按需启用，减少无关材料被提示词带偏的风险。

### 可沉淀表达

通过真实使用反馈识别 STT 分块策略带来的质量回退，将默认链路恢复为整段转录，同时保留可取消架构和诚实状态反馈。

## 2026-06-01｜STT 取消机制

### 问题

前端点取消只能中断请求，后端线程中的 faster-whisper 仍会继续占用 CPU。

### 判断

线程池无法可靠终止正在执行的推理任务。对于本地工具，用户点击取消应该真正释放计算资源。

### 方案

- 将 STT 从线程池迁移到独立子进程。
- 用户取消或 SSE 连接断开时，后端终止 STT 子进程。
- 任务终态记录为 `cancelled`，取消按钮点击仍作为用户行为事件单独记录。

### 取舍

牺牲跨任务模型热缓存的一部分速度优势，换取真正可取消和资源可控。

### 可沉淀表达

将本地 STT 推理改为可终止子进程架构，解决取消后 CPU 任务残留问题，提升长任务控制感和资源安全性。

## 2026-06-01｜长字幕笔记生成模式

### 问题

直接调用 DeepSeek API 生成课程笔记时，长字幕容易被选择性概括，产出的笔记不如将字幕交给 Codex 处理完整。

### 判断

问题不只是上下文长度，而是长内容下的质量策略。即使模型能接收较长输入，也可能丢失例子、推导过程和老师强调点。

### 方案

- 新增三种笔记生成模式：
  - `auto`：默认自动选择。
  - `direct`：整段长上下文直接生成。
  - `high_fidelity`：分段提取证据，再生成终稿，并尽量做覆盖率检查。
- 当前自动阈值约为 20,000 字符，超过后走高保真模式。
- 前端工作台支持选择模式，上传、字幕导入和重新生成都会传递 `note_mode`。
- 事件日志记录 `requested_note_mode`、`resolved_note_mode`、分段数和覆盖检查结果。

### 取舍

高保真模式更慢、更贵，但更适合课程笔记场景；直接模式保留给短材料和对照测试。

### 可沉淀表达

针对长字幕笔记质量下降问题，设计直接长上下文与高保真证据链两种生成策略，并通过事件 metadata 支持后续质量与耗时对比。

## 2026-06-01｜处理中心改为工作台

### 问题

原来的“处理中心”只在任务运行时展示流水线状态，空闲时几乎没有使用价值；真正高频调整的参数反而散落在设置页里，导致每次处理前需要进入低频设置区域修改。

### 判断

工具型产品应把“开始任务前必须确认的参数”放在离任务最近的位置。设置页应该保留低频凭证、模板维护和偏好，而不是承载日常运行决策。

### 方案

- 将导航中的“处理中心”改为“工作台”。
- 工作台集中承载高频运行参数：
  - 默认提示词模板
  - STT 模型、语言、速度档
  - AI 服务商、模型和 API Key
  - 笔记生成模式
  - 仅转录开关
  - 飞书自动导出与 lark-cli 导出方式
- 所有工作台参数仍写入同一份 `fluentflow_settings`，仪表盘上传、字幕导入、编辑器重新生成和重新转录继续读取同一配置。
- 设置页移除高频运行参数，只保留飞书凭证、提示词模板库、导出历史、主题和清除历史。

### 取舍

工作台变得更像“任务控制面板”，设置页更轻；代价是需要明确区分“选择当前运行默认值”和“维护模板内容”这两类行为。

### 可沉淀表达

将低价值的处理中心重构为运行工作台，把高频参数前置到任务开始前，减少设置页负担并降低每次处理前的认知成本。

## 2026-06-01｜前端运行方式生产化

### 问题

页面运行时仍依赖 `cdn.tailwindcss.com` 和浏览器内 Babel 转换 JSX，浏览器会提示不适合生产环境；这也让页面启动依赖运行时编译，前端形态不够稳定。

### 判断

当前项目可以继续保持单页 HTML + UMD React 的轻量形态，但不应该把 Tailwind 编译和 JSX 编译留到用户浏览器里。

### 方案

- 新增 `package.json`、`tailwind.config.cjs` 和前端构建脚本。
- 将原先内嵌在 `index.html` 的 JSX 抽到 `frontend/src/app.jsx`。
- 使用 Babel CLI 预编译为 `frontend/assets/app.js`。
- 使用 Tailwind CLI 生成本地 `frontend/assets/tailwind.css`。
- `index.html` 只加载本地构建产物，不再加载 Tailwind CDN 或 `@babel/standalone`。

### 取舍

增加了一个前端构建步骤，但保留了现有架构，没有迁移到 Vite/Next 等更重的工程体系。

### 可沉淀表达

将前端从浏览器运行时编译改为本地预构建产物，消除生产环境警告，提升页面加载稳定性，同时保持轻量单页部署方式。

## 2026-06-01｜STT 模型下限调整为 medium

### 问题

对比同一段求职经验交流录音的 `medium` 与 `small` 转录结果后，发现 `small` 在后半段出现大面积重复幻觉，例如长时间重复“进去,”和“就抢算,”，整体基本不可用。

### 判断

模型选项不能只按速度排序。对当前产品目标而言，可用性比速度更重要；如果 `small` 的结果需要大规模返工，它就不应该作为普通用户可选项。

### 方案

- 工作台 STT 模型下拉只保留 `medium（最低可用）` 和 `large-v3（最准确）`。
- 浏览器本地旧设置若仍是 `tiny`、`base` 或 `small`，打开工作台后自动升级为 `medium`。
- 后端收到空值或旧低配模型值时，同样兜底为 `medium`。
- STT benchmark 脚本默认模型改为 `medium`。
- 增加测试，确认旧低配模型会解析为 `medium`。

### 取舍

牺牲一部分低配模型的速度入口，换取默认结果的最低可用质量。未来如果要提供低配模型，应放在实验/高级选项里，并明确标注风险。

### 可沉淀表达

基于真实转录对比，将 STT 模型策略从“速度优先”调整为“最低质量门槛优先”，把 `medium` 设为产品可选下限，避免不可用结果进入主流程。

## 2026-06-01｜人工修正字幕沉淀为 STT 评测集

### 问题

单次人工校对如果只停留在字幕文件本身，价值会在当前任务结束后流失。后续更换模型、调整热词或增加纠错逻辑时，仍然只能凭主观感觉判断“有没有变准”。

### 判断

这份人工修正稿还不足以训练或微调 STT 模型，但足够成为第一条真实评测样本。它最适合用于衡量字符错误率、字幕段返工率、领域关键词召回和高风险错词是否减少。

### 方案

- 新增 `scripts/evaluate_stt.py`，支持 reference/hypothesis 字幕对比。
- 本地创建 `data/stt_eval/byte_job_share/`，存放人工修正稿、medium 转录稿、领域词表和错词对。
- 评测输出写入 `reports/stt_eval/byte_job_share/`，包括总体指标、差异字幕段、关键词召回和错词命中。
- 新增 `docs/stt_evaluation.md`，说明本地评测资产的目录结构和使用方式。

### 当前基线

- 字符级正确率：约 `95.806%`。
- 字幕段完全一致率：约 `78.706%`。
- 领域关键词召回：约 `47.619%`。
- 已知错词对命中：`22` 类。

### 取舍

评测资产放在 `data/` 和 `reports/` 下，不进入 Git，避免把私人字幕和报告内容提交到仓库。可复用的脚本、测试和说明进入项目代码。

### 可沉淀表达

将人工校对从一次性修字幕升级为 STT 质量基准，让模型选择、热词策略和纠错逻辑都有可复算的质量指标。

## 2026-06-01｜热词库与可选字幕审阅

### 问题

仅靠 STT 模型本身无法稳定识别求职、产品、数据分析和企业参访中的专有词；但让大模型直接改字幕又会带来过度润色、补充事实和误改风险。

### 判断

热词库应作为转录前的领域提示和转录后的审阅依据。大模型只能作为可选的保守复核器，不能拥有自由改写字幕的权限。

### 方案

- 新增内置热词库：通用、求职/面试、产品/数据、技术/AI、字节企业参访。
- STT 阶段将选中的热词库注入 faster-whisper 的 `initial_prompt` 和 `hotwords`。
- 新增字幕审阅模式：关闭、只标记建议、保守自动修正。
- 大模型审阅作为独立开关，只能接受或拒绝已有候选修正，不允许自由改写。
- 结果保留 `raw_transcript_text` 和 `transcript_text` 两份文本；生成笔记使用审阅后的 `transcript_text`。

### 取舍

默认不强制开启字幕审阅，避免把未验证的 AI 改写引入主流程。保守自动修正只处理高置信错词对，例如 `业务房 -> 业务方`、`AV实验 -> AB实验`；上下文相关项如 `自己 -> 字节` 只作为建议。

### 可沉淀表达

将人工校对中发现的术语错误转化为可复用热词库和保守审阅规则，用可选、可追溯、低权限的方式提升字幕质量。

## 2026-06-01｜STT 重复幻觉清洗

### 问题

转录结果中可能出现大量连续重复文字，例如“用户体验 用户体验 用户体验……”或连续多个字幕段重复同一句。这类内容会占用摘要上下文，并让大模型误以为重复词是高频重点。

### 判断

这是 STT 重复幻觉，不是普通术语错词。它适合用机械规则识别和折叠，不应该交给大模型自由改写。大模型可以做摘要，但不应该负责猜测重复区域真实说了什么。

### 方案

- 新增 `backend/core/transcript_cleaner.py`。
- 在音视频转写和字幕文件导入后，先执行重复清洗，再进入热词审阅和摘要生成。
- 清洗明显短语循环和连续重复字幕段。
- 保留 `raw_transcript_text` / `raw_segments`，新增清洗结果 metadata，摘要使用清洗后的文本。
- 前端编辑器展示“重复清洗”提示，说明折叠数量和重复单元。

### 取舍

只处理高置信、连续重复的机械噪声，不尝试恢复噪音区域的真实内容。这样牺牲了一部分激进修复能力，但能避免误删正常表达。

### 可沉淀表达

为本地 STT 增加重复幻觉检测与折叠层，将原始转录、清洗文本和摘要输入分离，减少模型 loop 对笔记质量的干扰。

## 2026-06-01｜SRT 字幕清洗与段落重组

### 问题

SRT/VTT 是面向播放器的字幕格式，包含时间码、序号和短字幕切片。即使模型上下文足够长，直接把原始字幕发给大模型，也会让大量时间信息占用上下文，并让模型更容易按字幕碎片而不是内容逻辑组织笔记。

### 判断

“导出字幕再发给大模型”和“在产品内生成笔记”的差异，不应该只体现在少一次复制粘贴。产品应该承担从播放格式到知识整理输入格式的转换：时间信息保留给预览、下载和时长统计；AI 输入则应该是更接近文章的干净正文。

### 方案

- SRT/VTT 解析继续保留原始 `segments`，用于时间轴预览、SRT/VTT 下载和统计。
- 发送给 AI 的 `text` 剥离时间码、字幕序号和 HTML 标签。
- 连续字幕按短停顿合并，遇到较长停顿、说话人切换或段落过长时分段。
- 中文片段合并时不强行加空格，英文片段保留必要空格。
- 增加解析器测试，覆盖中文合并、英文空格和长停顿分段。

### 取舍

当前实现是保守的段落归并，不把它伪装成真正的语义理解。真正的章节划分和重点组织仍交给摘要生成阶段的模型完成。

### 可沉淀表达

将 SRT/VTT 从播放字幕格式预处理为面向 AI 阅读的正文输入，减少时间码和碎片化字幕对上下文与笔记结构的干扰，强化产品内生成笔记相比手动投喂字幕的链路价值。

## 2026-06-02｜综合热词库与弹窗查看

### 问题

热词库数量不大，却被拆成多个可选预设，用户需要在开始转录前判断该选哪个库；“查看内容”以内联展开呈现，也会把工作台的参数区撑成说明区。

### 判断

热词库是系统能力，不应该让用户在普通使用中承担配置成本。当前更合理的产品形态是默认启用一个综合热词库，让用户只在需要确认内容时查看详情。

### 方案

- 前端工作台移除热词库下拉选择，改为单一“综合热词库”说明。
- 点击“查看内容”打开弹窗，集中展示包含来源、领域词和错词候选。
- 新请求默认使用 `all` 热词库；后端仍保留旧 preset key，兼容历史设置和旧请求。
- 后端未传热词参数时默认解析为综合热词库。

### 取舍

暂时不提供“不使用热词”的显性入口，因为当前热词库规模小、规则保守，配置选择带来的认知负担大于收益。若未来热词库明显扩大，或者出现热词误导转录的案例，再恢复高级开关。

### 可沉淀表达

将多预设热词配置收敛为默认综合库，降低转录前决策成本，同时保留可检查的热词与错词候选明细。

## 2026-06-02｜移除内置热词库

### 问题

内置热词库来自一次人工字幕审计，无法代表未来音视频材料。它看起来像通用能力，但实际收益集中在特定样本里的专有词和错词候选。

### 判断

热词库不应该成为默认产品能力。对普通材料来说，大模型摘要通常能容忍少量非关键错词；而错误或不相关的热词会增加配置成本，并可能给 STT 带来偏置。

### 方案

- 工作台移除热词库和基于热词的字幕审阅配置。
- 前端不再发送 `hotword_libraries`、`transcript_review_mode`、`transcript_review_use_ai`。
- 后端忽略旧客户端传入的热词和审阅参数，不再向 faster-whisper 注入热词。
- 保留重复幻觉清洗，因为它处理的是跨材料更稳定的机械噪声。

### 取舍

牺牲一部分针对特定样本的术语纠错能力，换取更低的用户认知负担和更稳定的普通材料转录。未来若恢复，应以文件级或项目级用户自定义术语的形式出现，而不是内置通用热词库。

### 可沉淀表达

删除不可泛化的内置热词能力，让 STT 主流程回到低配置、质量优先的稳定路径。

## 2026-06-02｜长任务后台化与可恢复进度

### 问题

音视频转录是长任务，不能把任务生命绑定在浏览器当前请求上。页面刷新、网络抖动或浏览器关闭如果直接取消后端计算，会让用户失去已经等待的时间；如果取消按钮又不能真正释放资源，则会造成 CPU 被无感占用。

### 判断

“连接断开”和“用户取消”是两个不同意图。前者应该允许任务继续运行并可恢复；后者才应该终止后台任务和 STT 子进程。

### 方案

- `/process` 上传后启动后台 Job，原 SSE 连接只订阅进度事件。
- 新增 `/jobs/{task_id}/events`，允许前端按任务 ID 重新订阅进度。
- 任务状态和结果继续写入本地 SQLite Job Store，刷新后可从 `/jobs` 恢复运行中的任务。
- 原始音视频保存到本地 `data/sources/{task_id}/`，支持历史结果不依赖浏览器文件权限重新转录。
- 前端启动时读取 running job，并重新订阅进度；完成后再写入历史。
- 取消按钮仍记录 `task_cancelled`，并显式取消后台任务，触发 STT 子进程清理。

### 取舍

当前后台任务是本机进程内任务，不是跨后端重启的持久队列。它解决浏览器断线和刷新问题；如果后端进程被杀，需要依靠已落库的最后状态和结果恢复，不能继续执行未完成推理。

### 可沉淀表达

将长任务从浏览器连接中解耦：断线可恢复，取消才终止，降低本地长转录任务的时间损失和资源失控风险。

## 2026-06-10｜演示级稳定性收口

### 问题

产品已经能完成上传、链接获取、转录、字幕和笔记生成，但演示或封闭 Beta 时最容易暴露的不是“缺少新功能”，而是失败信息不可读、产物入口不清晰、本地文件长期堆积，以及云端基础设施概念过早暴露给普通用户。

### 判断

当前阶段的优先级应该从继续扩展能力转向“让第一次完整流程更稳”。普通用户不需要理解 Azure、Blob、SAS、模型错误码；产品应该把这些复杂度吸收到后台，只在任务页告诉用户当前阶段、失败原因和下一步动作。

### 方案

- 后端新增统一错误翻译层，将常见 Azure、队列、文件过大、链接解析和本地模型错误转换为用户可理解的中文提示。
- 原始错误保留在任务 metadata 中，方便维护者排障，但不直接展示给用户。
- 后台任务页补充阶段详情，展示链接解析、视频下载大小、排队、完成和失败时的可行动信息。
- 结果产物区从灰色按钮升级为明确的下载入口，完成后显示 SRT、TXT、VTT、Markdown 和飞书文档链接。
- 本地产品介绍文档同步为“云端基础设施由产品维护者管理”，普通用户只看到上传/链接输入和本地/云端路线。
- Beta 清单保留本地清理脚本，明确默认 dry-run 和保留策略。

### 取舍

没有引入正式账号系统、云对象存储迁移或持久化队列。当前仍是适合封闭 Beta 和作品集演示的单机产品形态；如果进入公开多人使用，再补用户体系、任务隔离、额度、对象存储和队列服务。

### 可沉淀表达

把演示风险从“出现错误码就中断理解”降到“任务失败也能知道发生了什么”，让产品从功能可用推进到流程可展示、可排障、可验证。

## 2026-06-10｜云服务器试用形态收口

### 问题

如果把 FluentFlow 上线到云服务器给外部用户使用，继续暴露本地 faster-whisper、Azure Key、Blob/SAS、pyannote token、DeepSeek/OpenAI Key、飞书 App Secret 等配置，会让普通用户误以为自己需要理解和维护基础设施。

更大的风险是：本地转录路径在云服务器上会消耗主机 CPU/GPU，并且不同用户共享同一个服务进程；如果没有并发限制，少量长视频就可能拖垮封闭 Beta。

### 判断

公开试用形态应该把复杂度收回到产品维护者侧。普通用户只需要看到“上传文件/粘贴链接 → 后台转录 → 生成笔记/导出飞书”。云服务器第一版不追求多路线自由选择，优先保证体验口径稳定、资源消耗可控、失败原因可理解。

### 方案

- 新增 `FLUENTFLOW_PUBLIC_MODE=1` 运行模式。
- 公共模式默认只允许 `azure_batch` 转录；后端会把旧客户端传来的 `local` 兜底改为云端路径。
- 新增 `/runtime-config`，只向前端暴露运行模式、允许的 STT Provider 和任务限制，不暴露任何密钥。
- 前端根据运行配置隐藏维护者设置：本地模型、速度档、pyannote token、AI API Key、飞书 App ID/App Secret 等不再出现在普通用户工作台。
- 新增 `FLUENTFLOW_MAX_ACTIVE_JOBS_PER_CLIENT`，默认公共模式下每个设备最多 2 个排队/运行中的任务。
- 多文件上传、抖音链接任务和单文件处理都会在创建任务前做并发限制，避免单个浏览器无限堆长任务。
- Beta 部署清单补充云服务器推荐环境变量和口径：普通用户不理解 Azure，维护者负责配置 Speech、Blob/SAS、摘要模型和飞书导出。

### 取舍

这不是正式多租户架构。当前仍没有账号系统、用户级额度、对象存储生命周期策略或外部队列；`client_id` 只是封闭试用阶段的轻量隔离手段。它解决的是“能不能让别人顺利试用”，不是“能不能公开商业化开放”。

### 可沉淀表达

把云服务配置从用户任务流程中拿掉，让产品从“本地个人工具”推进到“可封闭试用的云端工作流产品”：用户只处理内容，维护者管理基础设施和资源边界。
