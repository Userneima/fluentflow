# FluentFlow | 视频课程笔记工作流工具产品复盘

## 一、产品基础信息

- **产品名称**：FluentFlow
- **产品定位**：面向长视频课程、课堂录音、讲座录音和会议录音的本地转写与 AI 笔记生成工作流工具
- **目标用户**：需要高频处理视频/音频学习资料，并希望沉淀到飞书知识库中的学生、研究生和知识工作者
- **核心价值**：把“视频/音频/字幕输入 → 本地转写 → AI 笔记化 → 飞书建档/本地下载”的多工具流程集中到一个产品中，降低多平台跳转、复制粘贴和格式整理成本
- **当前形态**：基于本地 FastAPI 服务和前端工作台运行，支持浏览器页面操作、飞书导出、本地历史记录和 SQLite 事件日志

## 二、产品诞生背景与核心痛点

考研和读研过程中，长视频课程、课堂录音和讲座录音经常持续几十分钟到数小时。传统处理方式通常是：

1. 先完整观看或反复回听内容。
2. 再用剪映等工具提取字幕。
3. 将字幕复制给大模型生成笔记。
4. 再复制到飞书中整理、归档和复习。

这个流程虽然比纯手工做笔记更快，但仍然存在明显问题：

- **工具割裂**：剪映、大模型、飞书之间频繁切换。
- **等待不可见**：长音频转写时无法判断是否卡死。
- **字幕不适合原样投喂**：SRT/VTT 中包含大量时间码和碎片化字幕行，会占用上下文，并让模型按字幕切片而不是按内容段落理解。
- **内容易丢失**：长字幕直接交给模型时，容易被选择性概括，漏掉例子、推导过程和老师强调点。
- **本地资源不可控**：早期取消任务后，后端转写线程可能继续占用 CPU。
- **数据不可回溯**：只知道“处理过很多文件”，但无法系统证明效率、稳定性和导出链路。

FluentFlow 的目标不是只做一个语音转文字工具，而是把长内容学习中的重复工作流产品化。

## 三、当前真实产品路径

### 1. 输入

用户可以在仪表盘中：

- 导入视频或音频文件，进入本地转写链路。
- 粘贴抖音分享文本或视频链接，系统先在后台解析并下载视频，再自动进入转写链路。
- 导入已有字幕/文本文件，跳过 STT，直接生成笔记。
- 在工作台中设置转录路线、语言、AI 服务商、提示词模板、笔记生成模式、仅转录模式和飞书导出方式。

普通用户不需要理解 Azure、Blob、SAS 或 Speech Key。云端转录所需基础设施由产品维护者在后端配置；用户只需要知道当前可以选择本地或云端转录路线。

### 2. 本地处理

本地 faster-whisper 路径会先通过 FFmpeg 抽取适合 STT 的 16kHz 单声道 WAV；Azure Batch 云转录路径会改用压缩 MP3 上传到 Blob SAS，减少上传体积并适配异步长音频处理。

当前产品不再提供内置热词库。早期热词来自单次人工字幕审计，不具备跨材料泛化能力；继续作为工作台配置会增加用户判断成本，并可能误导 STT。

STT 阶段会通过 SSE 返回进度和状态，包括：

- 加载 Whisper 模型
- 准备音频特征
- 等待第一段转录结果
- 接收转录片段
- 已转录到的音频时间

在进入摘要生成前，系统会先执行重复幻觉清洗。对于“用户体验 用户体验 用户体验……”这类明显短语循环，或连续多个高度重复字幕段，系统会折叠为更干净的文本，同时保留原始转录证据。

### 3. AI 笔记生成

转写或字幕解析完成后，系统会先把面向播放的字幕格式转换成面向阅读和 AI 理解的正文。生成笔记使用经过格式清洗和重复清洗后的 `transcript_text`；原始 STT 输出仍保留在 `raw_transcript_text` 中，方便追溯和对照。随后系统调用 DeepSeek / OpenAI 兼容接口生成结构化 Markdown 笔记。当前支持：

- 课程笔记
- 会议纪要
- 研究/论文摘要
- 快速要点提炼
- 自定义提示词

针对长字幕信息容易丢失的问题，新增三种笔记生成模式：

- **自动选择**：默认模式，约 20,000 字符以内直接生成，超过后自动进入高保真模式。
- **直接长上下文**：整段转录一次发送给模型，速度更快，适合较短材料或对照测试。
- **高保真笔记**：先分段提取证据，再生成终稿，并尽量做覆盖率检查，适合长课程和信息密度高的内容。

这里的产品价值不是把字幕文件简单转交给大模型，而是先在产品内完成格式清洗和段落重组：

- SRT/VTT 的时间码只保留在 `segments` 中，用于预览、下载和时长统计。
- 发送给 AI 的正文会剥离时间码、字幕序号和标签。
- 连续字幕会按停顿、说话人切换和段落长度重组成更接近文章的段落。
- 这样可以减少无关时间信息对上下文的占用，也避免模型被字幕切片干扰。

### 4. 飞书与本地导出

用户可以：

- 自动导出到飞书云文档。
- 在编辑器中手动导出到飞书。
- 下载转录文本、SRT、VTT、Markdown、PDF 或 Word 文档。
- 重新生成摘要，测试不同提示词和笔记生成模式。
- 在后台任务页查看每个长任务的状态、失败原因和已生成的 TXT/SRT/VTT/摘要产物。

## 四、核心功能模块

### 1. 音视频与字幕导入

- 支持常见视频、音频格式。
- 支持 SRT、VTT、TXT、MD 等已有字幕/文本文件。
- 支持粘贴抖音分享文本或视频直链，后台会尝试 direct、yt-dlp 和第三方解析策略获取 MP4，并保存来源 metadata。
- 字幕文件会跳过音频提取和 STT，避免伪造不存在的处理阶段。
- 对 SRT/VTT 字幕，产品会保留带时间轴的片段数据，同时生成一份去时间码、去标签、重组段落后的正文，作为 AI 笔记生成输入。

### 2. STT 转写：本地与 Azure Batch 云端路径

- 使用 FFmpeg 预处理音频。
- 本地路径使用 faster-whisper 完成语音转写，采用整段音频转录以保留上下文。
- 云端路径使用 Azure Batch + Blob/SAS，把 STT 阶段交给云端异步处理；Azure Fast inline 旧链路不再作为用户可见路线。
- Azure Batch 路径会先把音视频压缩成 MP3，上传到后端配置的 Azure Blob container SAS，再调用 Speech Batch API 提交任务、轮询状态、下载结果。
- Azure 路径仍复用同一套转录清洗、编辑器、AI 摘要和飞书导出链路；上传用 MP3，而不是本地 Whisper 专用 WAV。
- 云端配置状态只作为维护者排障信息，不应该成为普通用户的操作步骤。Dashboard 上传和编辑器重新转录会在上传前拦截未配置状态，后端也会在音频提取前做兜底预检。
- Batch 路径会展示上传 Blob、提交任务、等待 Azure、下载结果等状态；由于 Azure 异步排队不返回细粒度音频进度，产品不伪造 STT 百分比。
- 提供 `scripts/check_azure_batch_transcription.py` 做维护者使用的真实 Batch + Blob/SAS smoke test；提供 `--dry-run` 检查配置和音频预处理，不调用 Azure、不写事件表。
- 说话人区分按转录服务分流：本地路径使用 pyannote，Azure Batch 路径可请求云端 diarization 并使用返回的 speaker 标签；Azure Fast 旧链路不作为说话人区分主路径。
- Azure Batch 的“自动识别”会以 `zh-CN` 作为 fallback locale，并传入 `zh-CN` / `en-US` 候选语言；显式中文或英文会直接映射为对应 locale。
- 当前产品只开放 `medium` 和 `large-v3` 两个 STT 模型选项，其中 `medium` 是最低可用下限；旧的 `tiny`、`base`、`small` 设置会自动兜底为 `medium`。
- 不再支持内置热词库或基于热词的字幕审阅；当前优先保证普通材料的稳定转录和低配置成本。
- 本地路径支持速度档和模型设置；Azure 路径主要使用语言、云端凭证和可选说话人区分。
- 记录 STT 耗时、音频时长、实时倍率、转录服务、模型配置和设备环境。

### 3. 重复幻觉清洗

- 自动检测同一短语连续循环，例如 `用户体验 用户体验 用户体验...`。
- 自动检测连续多个高度重复的字幕段。
- 清洗层只做机械折叠，不补充事实、不润色、不猜测噪音区域真实内容。
- 摘要阶段使用清洗后的文本，避免重复内容占用上下文和干扰重点判断。
- 原始转录和原始 segments 始终保留，方便回溯。

### 4. 已移除：内置热词库与保守字幕审阅

- 早期内置热词库来自一次人工字幕审计，无法代表未来音视频材料。
- 热词对普通材料的收益不稳定，且会增加工作台配置负担。
- 当前主流程不再向 faster-whisper 注入热词，也不再基于热词生成字幕审阅候选。
- 后续如果恢复，只应做成文件级或项目级的用户自定义术语，而不是默认内置通用库。

### 5. 可取消的长任务机制

早期版本中，前端点击取消只能中断 HTTP 请求，后端线程中的 faster-whisper 仍可能继续占用 CPU。

当前版本已将 STT 迁移到后台任务和独立子进程：

- 用户取消任务时，后端会终止 STT 子进程。
- SSE 连接断开不再等同于取消任务，后台任务会继续运行。
- 前端可通过任务 ID 重新订阅进度并恢复结果。
- 原始音视频会保存到本地任务目录，历史任务可在不重新选择文件的情况下重新转录。
- 牺牲了一部分跨任务模型热缓存速度，换取真实可取消、可恢复和资源可控。

### 6. AI 结构化笔记生成

- 通过 DeepSeek / OpenAI API 生成 Markdown 笔记。
- 支持提示词模板和自定义提示词。
- 支持长字幕高保真生成策略，减少长内容被过度概括。
- 支持字幕正文预处理，避免将播放格式中的时间码、序号和碎片行原样交给模型。
- 记录摘要成功、失败、跳过和重新生成事件。

### 7. 飞书建档与下载

- 支持飞书 OpenAPI 或本地 lark-cli 导出。
- 导出结果记录飞书文档 URL。
- 支持摘要和转录文件的本地下载。
- 后台任务页会集中显示可下载产物，完成后可直接打开编辑器或下载 SRT/TXT/VTT/Markdown。

### 8. 后台任务、失败提示与本地清理

- 多文件或链接任务进入后台队列，不要求用户停留在开始页。
- 任务页展示解析链接、下载视频、上传云端、等待转录、生成摘要、导出飞书等阶段。
- 常见 Azure、链接解析、文件过大、队列调用失败会翻译成人能理解的失败原因，同时在 metadata 中保留原始错误便于排障。
- `scripts/cleanup_storage.py` 用于维护者定期清理本地源文件、产物、编辑记录和视频链接下载文件；默认 dry-run，确认后再 `--apply`。

## 五、数据能力与产品验证能力

FluentFlow 当前已经从“功能工具”扩展为“可观测工作流产品”。

### 1. 历史任务快照

通过前端 localStorage 和后端 uvicorn 日志，可以回溯已有使用规模，但这些数据只作为历史快照，不伪造成未来事件日志。

已生成本地报告：`reports/fluentflow_history_snapshot.md`。

### 2. SQLite 事件日志

新增本地事件日志：`data/fluentflow_events.sqlite`。

第一阶段只记录真实发生的系统链路和已有用户按钮行为，包括：

- 文件导入
- 音频提取完成
- STT 完成
- 重复幻觉清洗完成
- 转写文本就绪
- 摘要完成 / 失败 / 跳过
- 飞书导出开始 / 完成
- 摘要下载
- 转录下载
- 任务失败
- 任务取消
- 任务终态

事件日志遵守两个原则：

- 不保存完整转写文本或完整 Markdown，只记录长度、耗时、状态、错误原因和链接。
- 不记录当前产品中没有真实入口的行为，例如“人工校对完成”“评分”“笔记可直接使用”。
- 重复清洗事件只记录折叠数量、移除 segment 数和清洗前后长度，不保存完整字幕正文。

### 3. 设备相关性能口径

STT 耗时强依赖设备性能，因此事件日志会记录：

- 操作系统和 CPU 架构
- CPU 核心数
- faster-whisper / ctranslate2 / FFmpeg 版本
- STT 服务：本地 faster-whisper 或 Azure 云转录
- 说话人区分后端：pyannote 或 Azure diarization
- STT 模型、速度档和语言
- 转写耗时 / 原音频时长比例
- 模型加载耗时
- 冷启动或缓存状态

这让报告可以写成“在当前设备环境下”的真实表现，而不是把某台设备的速度误包装成产品普遍性能。

## 七、近期关键迭代

### 1. STT 进度反馈从“假百分比”变成“真实状态”

问题：转录阶段长时间停在 22%，用户无法判断是否卡死。

解决：增加 STT 内部状态反馈，让用户知道当前是加载模型、准备音频、等待第一段，还是正在产出 segment。

### 2. 取消机制从“断开请求”变成“终止计算任务”

问题：前端取消后，后端推理线程仍可能继续占用 CPU。

解决：将处理流程放入后台任务，STT 放入独立子进程；连接断开不取消任务，用户明确点击取消时才终止子进程。

### 3. 长字幕笔记生成从“直接总结”升级为“双模式策略”

问题：长字幕直接交给模型时，笔记容易漏掉具体例子、推导过程和老师强调。

解决：增加 `direct` 和 `high_fidelity` 两条路径，默认 `auto` 按转录长度选择，并记录生成模式 metadata，方便后续对比质量与耗时。

### 4. 字幕输入从“播放格式”转换为“阅读格式”

问题：SRT/VTT 是给播放器使用的格式，包含时间码、序号和短字幕切片。如果直接交给大模型，会浪费上下文，并可能让模型按字幕碎片而不是内容逻辑组织笔记。

解决：产品内解析字幕时保留原始时间段 `segments`，但传给 AI 的正文会剥离时间信息，并把连续字幕重组成更自然的段落。这也是 FluentFlow 相比“把导出的 SRT 文件直接发给大模型”的关键差异：产品承担了从播放格式到知识整理输入格式的转换。

### 5. 从“功能完成”转向“数据验证”

问题：项目早期只能描述功能，缺少可证明的效率和稳定性数据。

解决：补充历史快照、事件日志、导出脚本和指标口径文档，让后续真实使用能自然沉淀数据。

### 6. 处理中心改为工作台

问题：原“处理中心”空闲时价值低，而提示词、STT 模型、AI Key、仅转录和飞书导出等高频参数都挤在设置页中。

解决：将处理中心改为工作台，把高频运行参数前置；设置页只保留飞书凭证、提示词模板库、导出历史、主题和清除历史等低频内容。

### 7. 前端运行方式从 CDN/Babel 迁到本地构建

问题：页面依赖 `cdn.tailwindcss.com` 和浏览器内 Babel，会出现生产环境警告，也让 JSX 和 Tailwind 编译发生在用户浏览器中。

解决：新增前端构建脚本，将 JSX 预编译为 `frontend/assets/app.js`，Tailwind 预编译为 `frontend/assets/tailwind.css`，`index.html` 只加载本地构建产物。

### 8. STT 模型下限从 small 调整为 medium

问题：真实字幕对比显示，`small` 在约 30 分钟录音中出现长时间重复幻觉，后半段基本不可用；`medium` 虽有小错误，但整体可作为人工校对基础。

解决：工作台只保留 `medium（最低可用）` 和 `large-v3（最准确）`。浏览器旧设置和后端旧参数中的 `tiny`、`base`、`small` 都会兜底为 `medium`。

### 9. 增加常见热词库和可选字幕审阅

问题：真实字幕对比显示，`medium` 已达到可用下限，但仍会在专业词、企业词、求职语境词上出错。直接让大模型修改整份字幕又会带来过度改写和误补事实的风险。

解决：新增常见热词库，并把热词库用于两处：转写前提示 STT，转写后生成保守错词候选。字幕审阅成为可选项，用户可以关闭、只看建议，或允许系统只自动修正明显错误。大模型审阅也是独立开关，并被限制为“判断候选是否成立”，不能自由重写字幕。

### 10. 增加 STT 重复幻觉清洗

问题：转录文稿中可能出现大量连续重复文字，例如“用户体验 用户体验 用户体验……”，这会占用摘要上下文，并让 AI 笔记误判重点。

解决：新增重复幻觉清洗层，在摘要生成前自动折叠明显短语循环和连续重复字幕段。清洗只处理机械重复，不做语义改写；原始转录仍保留用于追溯。

### 11. 修正 STT 分块转录带来的质量回退

问题：为了解决转录进度长时间不动，曾将音频默认按 60 秒切块转录。实际使用中发现转录结果反而变差，原因是切块打断了 faster-whisper 对连续上下文的利用。

解决：默认主流程切回整段音频转录，保留子进程取消能力和 segment 级真实进度。第一段 segment 产出前继续展示状态和等待时间，不再为了进度条连续移动牺牲转录质量。

### 12. 移除内置热词库

问题：内置热词库主要来自一次人工校对后的字幕审计，不能代表未来音视频。继续作为产品能力，会让用户以为它具备通用价值，并增加开始转录前的配置成本。

解决：从工作台移除热词库和基于热词的字幕审阅入口。前端不再发送热词或审阅参数；后端即使收到旧参数，也不再向 STT 注入热词或执行热词审阅。底层历史模块暂时保留，作为后续评估文件级术语功能时的参考。

### 13. Azure 云转录从 Fast inline 转为 Batch + Blob/SAS

问题：Fast inline 链路在真实文件体量下出现 EOF、Broken pipe、`InvalidModel` 等不稳定问题，并且不适合作为长音频和说话人区分的主路径。

解决：新增 Azure Batch + Blob/SAS 路径。工作台保存 Speech 地址、Key 和 Blob 容器 SAS；后端先上传压缩 MP3 到 Blob，再提交 Batch 任务、轮询状态并下载结果。Fast inline 旧链路保留为兼容和诊断入口，不再作为正式云端主路径。

### 14. 增加 Azure Speech 基础自检

问题：Batch + Blob/SAS 只能解决“大文件如何交给 Azure”的问题，不能证明 Azure Speech 资源本身已经能转录。继续直接测试长音频会把资源可用性、Fast API、Blob/SAS、Batch 轮询混在一起，难以定位失败原因。

解决：新增基础 Speech 自检。系统用 macOS `say` 生成一小段英文测试语音，转成 Azure 短音频 REST API 要求的 16kHz 单声道 PCM WAV，再调用短音频识别接口。当前真实自检结果显示 eastasia Speech 资源和 Key 能成功识别短音频；后续才需要继续验证 Blob/SAS 与 Batch 文件链路。

## 八、技术栈与实现

- **后端服务**：Python + FastAPI
- **音视频处理**：FFmpeg
- **本地转写**：faster-whisper
- **AI 笔记生成**：DeepSeek / OpenAI API
- **实时反馈**：Server-Sent Events
- **飞书导出**：飞书 OpenAPI / lark-cli
- **本地事件日志**：SQLite
- **前端形态**：浏览器工作台，支持仪表盘、工作台、编辑器和设置页
- **前端构建**：Tailwind CLI 生成本地 CSS，Babel CLI 预编译 JSX，生产页面不再依赖 Tailwind CDN 或浏览器内 Babel

## 九、产品收获

FluentFlow 的价值不在于单点语音转文字，而在于把长内容学习链路压缩成一个可复用、可观测、可持续迭代的本地工作流产品。

从产品角度，它验证了：

- 高频个人工作流值得被工具化。
- 长任务必须提供真实可理解的系统反馈。
- 本地 AI 产品需要处理设备差异、资源占用和取消机制。
- AI 生成质量不能只依赖“更长上下文”，还需要面向任务设计生成策略。
- 字幕文件不是天然适合 AI 生成笔记的输入，产品需要把播放格式转换为阅读和理解格式。
- STT 输出不能直接等同于可用正文；重复幻觉需要在进入摘要前先被机械清洗。
- STT 优化不能依赖一次性人工审计产物伪装成通用能力；没有明确材料上下文时，少配置比多配置更可靠。
- 简历中的“数据能力”不应靠虚构指标，而应来自真实使用日志和清晰口径。
