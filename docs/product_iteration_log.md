# FluentFlow 产品迭代记录

本文档按产品模块记录 FluentFlow 的关键迭代。它回答的问题不是“某次发布改了哪些文件”，而是“某个产品部分为什么一步步变成现在这样”。

文档分工：

- `docs/changelog.md`：按时间记录发布、部署、数据和维护影响，方便回滚、上线和排查。
- `docs/product_iteration_log.md`：按产品模块记录改动、时间和当时判断，方便长期理解产品演化。
- `docs/product_growth_log.md`：保留阶段性、作品集或简历表达层面的产品判断沉淀。

## 记录规则

当一次修改影响用户流程、产品判断、数据语义、维护方式或后续迭代方向时，在对应模块下追加记录。只改 typo、局部文案或无产品含义的内部重构时，不需要写入本文档。

跨模块改动只在主影响模块写完整记录，其他模块如果需要，只写一句“参见 ...”，避免同一件事维护多份解释。

## 入口 / 上传 / 导入

### Unreleased｜链接转录主页反馈对齐

- 修改时间：Unreleased（原时间线未标注具体日期；上线归档时补具体日期）
- 改了什么：通过抖音分享文本或视频链接提交任务后，开始页不再立刻跳转到后台任务页，而是像文件上传一样显示当前任务状态和处理阶段。
- 为什么这么改：用户从入口页提交任务后，如果立刻被带到后台任务页，会丢失“我刚才提交的东西正在被处理”的连续反馈。
- 当时判断：链接任务和文件任务在用户心智里都是“开始处理一个材料”，反馈模型应该一致；后台队列是实现细节，不应该成为用户被迫理解的页面跳转。
- 用户影响：用户可以在开始页看到解析链接、下载视频、语音转录、AI 摘要和飞书导出等阶段。
- 维护影响：主页恢复后台任务时需要读取 `queue_options` 中的转录路线、模型、语言、摘要模式和导出选项，避免刷新后当前任务卡口径丢失。
- 验证方式：重新构建前端资源，并用链接任务确认开始页状态持续更新。
- 后续风险 / 待观察：多文件队列、视频链接和本地上传应继续收敛到一致的任务反馈语言。

## 工作台 / 任务参数

### 2026-06-01｜处理中心改为工作台

- 修改时间：2026-06-01
- 改了什么：将导航中的“处理中心”改为“工作台”，把默认提示词模板、STT 模型、语言、速度档、AI 服务商、模型、笔记模式、仅转录和飞书导出等高频参数集中到任务开始前。
- 为什么这么改：原“处理中心”只在任务运行时有价值，空闲时几乎不能帮助用户做决策；真正高频的运行参数反而散落在设置页。
- 当时判断：工具型产品应该把“开始任务前必须确认的参数”放在离任务最近的位置，设置页只保留低频凭证、模板维护和偏好。
- 用户影响：每次开始处理材料前，可以在同一处确认关键运行参数，减少来回进入设置页的负担。
- 维护影响：所有工作台参数仍写入 `fluentflow_settings`，仪表盘上传、字幕导入、编辑器重新生成和重新转录继续读取同一配置。
- 验证方式：确认上传、字幕导入、编辑器重新生成和重新转录都使用同一套工作台设置。
- 后续风险 / 待观察：需要持续区分“当前运行默认值”和“维护模板内容”，避免工作台再次膨胀成设置页。

## 转录 STT

### 2026-06-03｜Azure 云转录从 Fast inline 转向 Batch + Blob/SAS

- 修改时间：2026-06-03
- 改了什么：将 Azure 云端长音频转录从 Fast inline 路径迁移为 Batch + Blob/SAS 异步路径，并加入 Blob SAS 配置、任务轮询、结果下载和说话人区分能力。
- 为什么这么改：Fast inline 在真实文件体量下出现 HTTPS 上传 EOF/Broken pipe、`InvalidModel` 和说话人区分能力不足等问题。
- 当时判断：长音频云转录应该走异步 Batch，而不是同步 inline；配置成本更高，但更符合长录音、大文件、可等待、可轮询的真实产品场景。
- 用户影响：云端转录更适合长材料，等待阶段更可解释，也能支持说话人区分。
- 维护影响：需要维护 Speech 资源、Blob container SAS、Batch 提交和轮询链路；真实 smoke test 会产生 Azure Storage 与 Speech 调用痕迹。
- 验证方式：使用 `scripts/check_azure_batch_transcription.py` 做 dry-run 或在配置齐全后做真实 Batch smoke test。
- 后续风险 / 待观察：Blob/SAS 配置仍有理解成本，公开使用前需要更清晰的配置自检和错误恢复。

### 2026-06-01｜STT 质量回退修正

- 修改时间：2026-06-01
- 改了什么：默认 STT 路径从 60 秒小块转录恢复为整段音频转录，分块能力保留在底层但不作为主流程；热词库改为用户明确选择时才注入。
- 为什么这么改：切块虽然让进度更容易推进，但会打断中文讲座、访谈和经验分享这类连续语境，导致上下文、句子延续和术语判断变差。
- 当时判断：对当前产品来说，转录质量比进度条连续移动更重要；如果进度收益换来大面积返工，就不应该作为默认策略。
- 用户影响：长材料转录质量更稳定，但第一段 segment 产出前仍可能等待较久。
- 维护影响：进度反馈要诚实表达模型状态，不能用分块制造虚假的顺滑感。
- 验证方式：对比同一材料的分块转录与整段转录质量，重点看连续语境、术语和重复幻觉。
- 后续风险 / 待观察：如果未来重新启用分块，需要有评测集证明质量没有回退。

### 2026-06-01｜STT 取消机制

- 修改时间：2026-06-01
- 改了什么：将 STT 从线程池迁移到独立子进程，用户取消或 SSE 连接断开时，后端终止 STT 子进程。
- 为什么这么改：前端中断请求不能停止线程池里的 faster-whisper 推理，取消后 CPU 仍会继续被占用。
- 当时判断：对本地工具来说，用户点击取消应该真正释放计算资源；线程池无法可靠终止正在执行的推理任务。
- 用户影响：长任务可以被真正取消，不会继续拖慢本机。
- 维护影响：牺牲一部分跨任务模型热缓存速度，换取资源可控和任务终态明确。
- 验证方式：启动长音频转录后取消，确认任务终态为 `cancelled`，并确认推理子进程退出。
- 后续风险 / 待观察：后续如果引入并发队列，需要继续保证取消不会留下后台推理残留。

### 2026-06-01｜STT 模型下限调整为 medium

- 修改时间：2026-06-01
- 改了什么：工作台 STT 模型下拉只保留 `medium` 和 `large-v3`；旧的 `tiny`、`base`、`small` 设置会自动升级到 `medium`。
- 为什么这么改：真实求职经验交流录音对比中，`small` 在后半段出现大面积重复幻觉，整体不可用。
- 当时判断：模型选项不能只按速度排序；当前产品目标下，可用性比速度更重要。
- 用户影响：普通用户不会再误选明显低于质量门槛的模型。
- 维护影响：后端和前端都要兜底旧低配模型值，benchmark 默认模型也改为 `medium`。
- 验证方式：测试旧低配模型值解析为 `medium`，并用真实样本确认质量门槛。
- 后续风险 / 待观察：如果未来提供低配模型，应放在实验或高级选项，并明确标注风险。

## AI 摘要 / 笔记生成

### Unreleased｜策略 Agent 推荐处理方案

- 修改时间：Unreleased（原时间线未标注具体日期；上线归档时补具体日期）
- 改了什么：编辑器 AI 摘要区域新增“推荐方案”，根据文件名、转录稿预览、文本长度、音视频时长和可选目标推荐材料类型、笔记模式和提示词模板。
- 为什么这么改：用户面对不同材料时，不一定知道应该选哪种笔记模式或模板；把判断压力完全丢给用户会增加使用门槛。
- 当时判断：推荐可以降低认知负担，但不能自动改写当前摘要；用户需要明确点击“应用到下次生成”后才生效。
- 用户影响：用户获得一个可接受、可拒绝的处理建议，而不是被系统静默改变设置。
- 维护影响：新增 `POST /plan-note-task` 和 `backend/core/note_planner.py`，推荐结果必须经过白名单校验，不内置模型。
- 验证方式：确认 `agent_plan_generated`、`agent_plan_failed`、`agent_plan_accepted` 事件记录正确，且无 API Key 时不影响主流程。
- 后续风险 / 待观察：采纳率只能解释为“用户是否采用推荐处理方案”，不能直接代表最终笔记质量。

### Unreleased｜长字幕完整覆盖笔记模式

- 修改时间：Unreleased（原时间线未标注具体日期；上线归档时补具体日期）
- 改了什么：工作台新增 `完整覆盖笔记` 模式，先提取证据 ID，再生成章节大纲、归类证据、按章节生成笔记，并检查高重要性证据是否遗漏。
- 为什么这么改：长字幕直接摘要容易选择性概括，课程、讲座等材料需要更强的覆盖意识。
- 当时判断：完整覆盖模式更慢、更贵，不应直接成为 `自动选择` 的默认路径，避免用户突然承担等待时间和模型调用成本。
- 用户影响：用户可以手动选择更适合长材料的完整覆盖笔记。
- 维护影响：`backend/core/ai_summarizer.py` 新增 `chapter_coverage` note mode，并记录证据数、章节数、高重要性证据数和覆盖数。
- 验证方式：按 `docs/note_mode_evaluation_plan.md` 跑离线评测，再公开扩大使用。
- 后续风险 / 待观察：流程覆盖指标不等于用户满意度，仍需要真实材料评测和用户反馈。

### Unreleased｜笔记内容与格式提示词分离

- 修改时间：Unreleased（原时间线未标注具体日期；上线归档时补具体日期）
- 改了什么：将笔记模板拆分为内容模板和格式模板；内容模板决定覆盖什么信息，格式模板决定标题序号、表格、代码块、段落等飞书 Markdown 样式。
- 为什么这么改：把内容策略和格式偏好混在同一段提示词里，会让用户难以只调整样式或只调整内容。
- 当时判断：内容规则和格式偏好应该分层，避免格式诉求污染内容判断。
- 用户影响：用户可以在编辑器模板面板和设置页查看、编辑、恢复默认格式模板。
- 维护影响：摘要、重新生成、字幕导入和队列处理链路都新增 `format_prompt` 参数。
- 验证方式：确认不同入口都能传递格式模板，并且历史任务不会被错误回填当前设置。
- 后续风险 / 待观察：暂不引入独立 skill 系统，避免模板体系过早复杂化。

## 编辑器 / 转录修正

### 2026-06-01｜人工修正字幕沉淀为 STT 评测集

- 修改时间：2026-06-01
- 改了什么：新增 `scripts/evaluate_stt.py`，将人工修正稿、medium 转录稿、领域词表和错词对组织成本地 STT 评测资产。
- 为什么这么改：一次人工校对如果只停留在字幕文件本身，价值会在当前任务结束后流失。
- 当时判断：这份修正稿不足以训练模型，但足够成为真实评测样本，用来衡量字符错误率、字幕段返工率、关键词召回和高风险错词。
- 用户影响：短期无直接界面变化，长期能让模型选择和热词策略更稳。
- 维护影响：评测资产放在 `data/` 和 `reports/` 下，不进入 Git；可复用脚本和说明进入仓库。
- 验证方式：运行 STT 评测脚本，输出总体指标、差异字幕段、关键词召回和错词命中。
- 后续风险 / 待观察：评测样本数量仍少，不能把单一材料结论泛化到所有场景。

## 历史记录 / 账号同步

### Unreleased｜本地 App 支持本机转录与云端账号历史分离

- 修改时间：Unreleased（原时间线未标注具体日期；上线归档时补具体日期）
- 改了什么：本地 App 连接云端账号工作区时，仍可显示本机可用的“本地转录”选项；本地 STT 完成后把转录文本和摘要同步进当前云端账号历史。
- 为什么这么改：本地桌面和云端网页的设备能力不同，不能因为账号工作区连接云端，就把本机可用能力隐藏掉。
- 当时判断：账号历史可以统一，但执行能力应该按当前运行环境判断；本地页面和云端页面可以有不同设备能力状态。
- 用户影响：本地 App 用户可以继续使用本机转录，同时把结果沉淀到账号历史。
- 维护影响：云工作区模式下 `/runtime-config`、`/health`、`/credentials/status`、`/speaker-diarization/status` 保持本地响应；本地 STT 单文件任务通过 `X-FluentFlow-Execution-Target: local` 绕过云端代理。
- 验证方式：重建前端资源并重启本地后端，确认本地 STT 和云端账号历史同步都正常。
- 后续风险 / 待观察：第一版优先覆盖单文件上传和编辑器重新转录；视频链接已补本地执行头链路，多文件后台队列仍需继续收敛。

### Unreleased｜账号历史长期保留与本机历史一次性导入

- 修改时间：Unreleased（原时间线未标注具体日期；上线归档时补具体日期）
- 改了什么：登录账号后的历史记录不再被自动按条数或天数删除；本机旧历史导入到账号后，不会反复提示同一批记录。
- 为什么这么改：账号历史和本机临时历史的保留语义不同，账号历史如果被后台保留策略裁剪，会破坏用户对“账号资产”的预期。
- 当时判断：账号作用域 `client_id = user:{account_id}` 的记录应跳过本地历史自动裁剪；导入提示也应按账号记忆处理状态。
- 用户影响：登录后的历史记录更稳定，旧本机历史不会重复打扰用户导入。
- 维护影响：导入候选去重同时使用 `taskId`、`originalTaskId`、`sourceFingerprint`，并新增按账号保存的导入处理状态。
- 验证方式：重建前端资源并重启后端，确认账号历史不会被保留策略删除，导入提示不会重复出现。
- 后续风险 / 待观察：旧浏览器 `localStorage` 历史不会立即删除，只会在候选检测中被识别为已处理。

### Unreleased｜本地桌面与云端账号历史同步

- 修改时间：Unreleased（原时间线未标注具体日期；上线归档时补具体日期）
- 改了什么：登录后检测到本机旧历史时，在开始页提示用户手动确认导入；桌面快捷方式默认把账号、任务、上传、历史和额度 API 转发到 `https://fluentflow.icu`。
- 为什么这么改：桌面和网页需要逐步使用同一份云端账号历史，但旧本机历史包含用户材料结果，不能静默上传。
- 当时判断：历史同步必须把隐私边界放在前面；只上传转录文本和摘要，不上传原始音视频，且需要用户确认。
- 用户影响：用户可以把旧本机历史迁移进当前账号，并在云端账号历史中继续查看。
- 维护影响：新增 `POST /account/import-history` 和 local-only `GET /local-history/candidates`，导入记录生成新的 `imported_*` task id。
- 验证方式：重新构建前端资源，更新桌面 `FluentFlow.app` 快捷方式，确认导入候选、确认导入和历史显示正常。
- 后续风险 / 待观察：重复导入依赖原始 task id 和 source fingerprint，后续跨设备迁移仍需关注去重可靠性。

## 额度 / 权限 / 管理员

### Unreleased｜管理员额度与历史导入口径修复

- 修改时间：Unreleased（原时间线未标注具体日期；上线归档时补具体日期）
- 改了什么：管理员侧栏和管理页额度显示为“额度豁免 / 无限”；导入本机历史不再占用每日处理上限；管理员不受普通账号/设备每日任务数和上传量限制。
- 为什么这么改：管理员和普通用户的额度语义不同，把管理员显示成普通余额数字会误导维护判断；历史导入也不是用户当天提交的新处理任务。
- 当时判断：额度、任务提交量和历史迁移必须分开计数，否则后续收费、风控和运营判断都会混乱。
- 用户影响：管理员看到更准确的额度状态，普通用户历史导入不会污染每日处理上限。
- 维护影响：`/account/quota` 和 `/admin/users` 返回 `unlimited`、`quota_exempt`，每日任务统计排除 `imported_local_history`。
- 验证方式：重新构建前端并重启后端，确认管理员额度显示和导入统计口径。

- 维护影响：事件不保存完整转写文本或完整 Markdown，只保存长度、耗时、状态、错误原因和链接。
- 验证方式：运行事件导出脚本，检查 JSON / CSV 中的事件类型和 metadata 口径。
- 后续风险 / 待观察：完成导出不等于认可质量，下载摘要不等于笔记可用，指标解释必须保守。

- 验证方式：生成历史快照报告并检查来源、可信度和可用口径。
- 后续风险 / 待观察：历史快照不能和真实事件日志混算。
