# Product Portfolio Section Revision Guide

## 文档名称

产品作品集项目页优化说明

## 适用场景

用于优化面向面试官的产品作品集文档。目标不是完整记录项目历史，而是让面试官快速判断：

- 这个产品是什么
- 解决了什么真实问题
- 核心流程如何工作
- 做了哪些关键产品取舍
- 当前结果是否可验证
- 候选人在其中承担了什么角色

## 总原则

1. 产品优先，个人信息后置。
   开头不要先写长篇个人介绍。先让读者理解作品集里有哪些产品，以及每个产品为什么值得看。

2. 少讲技术堆栈，多讲产品判断。
   技术只作为实现手段出现，不作为项目介绍主线。

3. 每个项目只保留证据链。
   面试官最需要看到的是：问题、方案、流程、取舍、结果、证据。

4. Demo 视频放在项目正文前半段。
   视频是最强证据，应尽早出现，而不是放在最后当附件。

5. 不展示普通用户不该理解的配置项。
   API Key、云服务配置、SAS URL、后台基础设施等内容不应成为作品集重点。

## 推荐整页结构

```markdown
## 一、作品集定位
说明这是产品作品集预览页，用于快速展示几个真实项目。

## 二、项目总览
用一张表概括所有项目。

## 三、项目一：项目名｜一句话定位
### 1. 产品是什么
### 2. Demo 与预览
### 3. 解决的问题
### 4. 核心流程
### 5. 关键产品取舍
### 6. 当前结果

## 四、项目二：项目名｜一句话定位
同上。

## 五、项目三：项目名｜一句话定位
同上。

## 六、个人角色与能力说明
把个人贡献、方法论和能力总结放在末尾。

## 七、文档说明
说明素材待补充、项目范围和使用场景。
```

## 项目段落模板

```markdown
## 项目名｜一句话定位

### 1. 产品是什么
用 2-4 句话说明产品，不要写成技术方案。

### 2. Demo 与预览
- 演示视频：
- 产品 Demo 链接：
- 结果产物示例：
- 项目说明 / 仓库：

### 3. 解决的问题
说明原流程为什么低效，用户为什么会需要这个产品。

### 4. 核心流程
用 4-6 个步骤描述用户如何完成任务。

### 5. 关键产品取舍
只写最能体现产品判断的 3-5 个取舍。

### 6. 当前结果
写可验证结果：效率变化、真实使用量、产物样例、测试轮次、截图或链接。
```

## 每个项目应该保留的内容

- 一句话定位
- 用户问题
- 核心流程
- 关键取舍
- Demo 视频 / 预览链接
- 可验证结果
- 个人贡献，但放在后面或压缩表达

## 每个项目应该删减的内容

- 过长背景故事
- 没有证据支撑的价值判断
- 技术名词堆叠
- 开发过程流水账
- 普通用户不会接触的配置细节
- 和产品判断无关的个人介绍

## FluentFlow 示例重点

FluentFlow 应该强调：

- 它不是普通视频转文字工具，而是视频/录音到学习笔记的完整工作流。
- 核心流程是：上传音视频/字幕 → 后台转录 → 字幕可编辑 → 偏好化笔记 → 飞书沉淀。
- 演示视频要重点呈现后台队列、完整转录、字幕编辑、笔记偏好和飞书导出。
- Azure、API Key、SAS URL 等基础设施不应在作品集中重点出现。

## 面试官视角检查清单

修改后检查：

- 读者 30 秒内能否知道这个产品是什么？
- 读者 1 分钟内能否知道它解决什么问题？
- 是否有 Demo 视频或结果证据？
- 是否能看出产品取舍，而不是只看见功能列表？
- 是否避免了过多个人介绍抢占首屏？
- 是否避免了配置项和技术细节干扰阅读？
