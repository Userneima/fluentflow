# FluentFlow UI / Dark Mode Design System

本文档把 FluentFlow 的界面设计方法固定下来，尤其用于暗黑模式、任务入口、结果编辑、设置、记录列表、处理详情和管理类页面。目标不是做一套炫技视觉，而是让长期维护时每个页面都能保持清晰、稳定、可读、可扩展。

## Product Read

FluentFlow 是一个长任务工具产品。用户目标是上传音视频、等待后台转录、编辑字幕、生成笔记、查看历史和管理账号额度。

默认设计判断：

- 页面类型：工具型产品，不是营销落地页。
- 视觉目标：安静、可信、信息密度适中，支持反复使用。
- 交互目标：核心动作明确，后台状态可追踪，错误可理解。
- 暗黑模式目标：降低视觉疲劳，但不能牺牲可读性和操作识别。

设计参数：

- `DESIGN_VARIANCE: 4`
- `MOTION_INTENSITY: 2`
- `VISUAL_DENSITY: 7`

含义：布局可以有轻微层次，但不做大幅艺术化；动效只服务状态反馈；信息密度要高于普通官网，但不能变成拥挤后台。

## Typography Constraints

先定字体层级，再做页面。

### 允许 display-scale 的位置

- 主入口页或任务入口页的核心行动标题。
- 空状态的核心行动提示。
- 少量产品介绍或演示入口。

### 必须使用 operational-scale 的位置

- 结果编辑页标题、字幕列表、摘要面板、记录列表、设置页、账号面板、管理页。
- 表单标签、输入框、按钮、卡片标题、表格行、状态徽标。

### 默认字号上限

| 场景 | 推荐上限 |
| --- | --- |
| 工具页主标题 | `text-3xl` 到 `text-4xl` |
| 面板标题 | `text-lg` 到 `text-2xl` |
| 卡片标题 | `text-base` 到 `text-xl` |
| 表单标签 | `text-xs` 到 `text-sm` |
| 按钮文字 | `text-sm` 到 `text-base` |
| 表格、记录列表、元信息 | `text-xs` 到 `text-sm` |
| 标题旁状态标签 | `text-[10px]` 到 `text-xs`，必须小于标题字号 |
| 字幕正文 | `text-base` 到 `text-lg`，按阅读距离调整 |

不要用 viewport width 驱动工具页文字大小。需要 `clamp()` 时必须有明确上限，避免宽屏下控件和面板标题膨胀成海报字。

## Page Header Density

FluentFlow 的产品页顶部是操作型页面标题，不是营销 hero。除任务入口页的核心上传/导入入口、空状态的首次行动提示、或明确的对外介绍页外，不要在应用内页面顶部放大面积品牌标题、宣传语或解释性段落。

默认规则：

- 页面 header 应该帮助用户定位当前位置、看到当前状态、进入关键动作。
- 标题通常使用 `text-xl` 到 `text-2xl`；只有真正的页面主入口可到 `text-3xl`。
- 说明文字最多一到两行，优先回答“这里能做什么”，不要重复解释产品理念。
- Header 垂直空间应尽量控制在一个紧凑工具栏或小型页面标题区内，让任务、表单、列表、编辑内容尽早出现在首屏。
- 如果一段内容不能改变用户下一步动作，不要让它占据首屏空间。放到具体卡片、空状态、折叠详情、帮助文档或设置说明里。
- AI、自动化和处理详情相关页面尤其要避免“看起来智能”的大标题区。智能能力的价值应通过真实计划、判断依据、状态和恢复动作呈现。

## Secondary Surface Consistency

登录页、访问门禁、关于与协议、设置里的维护区、提示词模板弹窗、确认弹窗和空状态也属于产品体验，不是可以套旧模板的边缘页面。

默认规则：

- 低频子页面也要使用当前产品的中性工作流语言：统一的主操作、边界、输入背景、圆角和暗黑模式表面层级。
- 不要让低频页面退回旧视觉语言。旧语言的常见症状包括过小圆角、无语义的紫色按钮、slate 风格输入框、厚重默认阴影或泛 SaaS 卡片；这些是排查线索，不是永久禁用某个颜色或类名。
- 登录和访问门禁页不做营销 hero。它们的主任务是验证身份，首屏只保留产品身份、当前动作、必要说明、表单、错误反馈和访客入口。
- 关于与协议页不做大面积品牌卡片。使用紧凑页面标题和按主题排列的条目，让用户快速理解边界、数据、账号和责任。
- 弹窗里的颜色、圆角、输入框、按钮必须和主产品页一致。不要因为弹窗是局部组件就回到旧组件库风格。
- 设置页只维护长期偏好、凭证和本机数据。单次任务判断、处理原因和生成详情不放回设置页。

落地检查：

- 检查目标子页面是否仍有旧视觉语言残留，例如不一致的圆角、无语义强调色、旧输入框、厚重阴影或默认组件库痕迹。
- 每个表单控件是否有可见 label，不能只靠 placeholder。
- 每个高风险动作是否有明确确认或足够降级的视觉权重。
- 子页面是否和侧边栏、处理详情页、结果编辑页使用同一套按钮、输入框和卡片语言。

## Page Task And Information Architecture

每个页面先定义主任务，再设计布局。主任务不是“这个页面有哪些功能”，而是用户进入页面时最需要完成的一件事。例如结果编辑页的主任务是复查、修改和导出结果；处理详情页的主任务是解释任务经过、判断依据和失败恢复；设置页的主任务是维护长期偏好和凭证。

默认规则：

- 每个页面只能有一个主任务。直接服务主任务的内容和操作放在默认视图，间接服务的能力收纳到菜单、折叠详情、浮层或二级页面。
- 页面信息先分成四类：内容、状态、操作、解释。内容是主体；状态辅助理解；操作推动任务；解释用于追溯原因。四类信息不要混在同一个视觉区域里。
- 默认视图服务 80% 使用场景。少数情况下才需要的高级信息，不要常驻占用核心内容空间。
- 内容优先于解释。用户进入工具页通常是为了处理对象、查看列表、编辑正文或导出结果，不是阅读系统说明。
- 用户语言优先于系统语言。模块标题应描述用户看到的对象和可执行动作，例如“笔记正文”“转录原文”“复查结果”，不要优先暴露内部实现，例如“AI 摘要”“生成链路参数”。
- 高频、低风险动作可以直接可见；低频或高风险动作应降级、收纳或二次确认。按钮大小、位置和颜色必须反映操作优先级。
- 解释可以存在，但默认不能压缩核心内容。处理判断、模型参数、生成原因、失败诊断等，应优先放在处理详情页；只有当解释会直接改变当前页面动作时，才使用本页折叠详情或浮层。
- 结果编辑页只承载复查、修改、下载和导出。提示词模板、生成模式、分块数、摘要依据、模式原因等生成详情，不要常驻或折叠展示在笔记正文区域；保留一个进入处理详情页的入口即可。
- 历史/记录列表页主要承载过去成功任务、失败记录、打开结果、下载、删除和处理详情入口；进行中任务只在存在时临时置顶或显示筛选，不要长期占据一级结构。
- 失败记录的主动作优先是“重新处理”或“查看详情”，不要把不可打开的失败结果做成醒目的主按钮。
- 历史/记录列表页可以展示失败原因摘要，但摘要生成诊断、下一步解释不要常驻挤压记录列表；用户需要追溯原因时进入处理详情页。
- 用户侧一级入口使用“处理记录”，统一承载进行中任务、失败记录、历史结果、下载产物和处理详情；不要在侧边栏同时保留“历史记录”和“处理记录”。“历史记录”只用于说明本机浏览器缓存或旧数据兼容，“后台任务”只用于解释运行机制、排障文档或后端实现。
- 历史/记录列表页可以先展示缓存以减少等待，但不能把缓存伪装成最终真实状态；后台刷新后必须合并真实状态，刷新失败时保留缓存并给出轻量提示。
- 处理详情页不是通用流水线日志。默认主体应展示当前素材特有的判断：材料类型、判断依据、路线选择、笔记策略、对结果的影响和复查点。通用执行步骤只作为辅助记录。
- 处理详情页的进度状态优先由顶部核心概览承载；不要在主内容区再新增大面积“接收材料、转录、生成、保存”步骤列表，除非它替代了概览且能展示当前素材特有的失败原因或判断依据。
- 处理详情页展示整理后的可复查依据，不展示模型原始内心独白。推荐结构是“判断结论 / 依据 / 对结果的影响 / 下一步”，而不是长段自述。
- 如果某个信息块不能改变用户当下动作，也不能帮助用户判断当前内容是否可信，就不要放在主内容区。

落地检查：

- 页面第一屏是否能看到主任务对象，而不是欢迎语、产品理念或低频参数。
- 主按钮是否对应页面主任务，而不是技术上最容易暴露的能力。
- 状态徽标、元信息和解释性文字是否正在挤压正文、列表、表单或编辑区。
- 低频操作是否和高频操作同级竞争，例如把“重新转录”和“导出”做成同样大的按钮。
- 内部术语是否已经转译成用户可理解的对象、状态或行动。

## Theme Model

FluentFlow 使用 Tailwind `darkMode: "class"`，并通过 CSS 变量提供语义色。语义 token 是主题的事实来源。

优先使用这些类：

- 背景：`bg-surface`、`bg-surface-dim`、`bg-surface-container-lowest`、`bg-surface-container-low`、`bg-surface-container`、`bg-surface-container-high`
- 文本：`text-on-surface`、`text-on-surface-variant`、`text-on-background`
- 边框：`border-outline`、`border-outline-variant`、`ff-border-muted`、`ff-border-control`
- 重点动作：`bg-primary text-on-primary` 或 `text-primary`
- AI 摘要或次级智能动作：`tertiary` 系列，但要克制使用

不要新增长期存在的原始颜色工具类作为主界面表面。下面是常见风险示例，具体判断以是否破坏语义 token 和主题一致性为准：

- `bg-white`
- `bg-black`
- `bg-slate-*`
- `bg-gray-*`
- `text-gray-*`
- `text-slate-*`
- `bg-blue-50`

如果只是局部迁移期间保留旧类，可以接受，但改到相关区域时应该顺手替换为语义 token。

## Surface Stack

暗黑模式的层次靠表面和边框，不靠强阴影、发光和大面积渐变。

| 层级 | 用途 | 推荐类 |
| --- | --- | --- |
| Page shell | 页面底层背景 | `bg-surface text-on-surface` |
| Sidebar / app chrome | 侧栏、顶部导航、固定导航 | `bg-surface-container-lowest` |
| Main workspace | 任务工作台、结果编辑主体区域 | `bg-surface-container-low` |
| Panel / section | 转录面板、摘要面板、设置分组 | `bg-surface-container-lowest ff-border-muted` |
| Nested control row | 工具栏、筛选条、输入组 | `bg-surface-container-low ff-border-control` |
| Active / selected | 当前菜单项、选中任务、当前步骤 | `bg-primary/10 text-primary` |
| Popover / modal | 弹窗、菜单、浮层 | `bg-surface-container-lowest ff-border-muted shadow-sm` |

不要把卡片套卡片。页面分区用宽度、间距和分隔线组织；卡片只用于重复条目、弹窗、真正需要边界的工具面板。

## Text Hierarchy

| 信息类型 | 推荐类 |
| --- | --- |
| 主要内容 | `text-on-surface` |
| 次级说明、时间、来源、状态描述 | `text-on-surface-variant` |
| 弱化信息、禁用态 | `text-outline` 配合透明度 |
| 时间戳、额度、数量、耗时 | `tabular-nums`，必要时用 `font-mono` |
| 错误 | `text-error` 或错误容器 token |

不要用低对比度灰色承载关键状态。暗黑模式里“看起来很淡”不等于高级，状态和操作必须能一眼识别。

## Action-Oriented Copy

FluentFlow 是工作流工具。标题、按钮、下一步提示和流程阶段应优先引导用户行动，而不是只描述系统状态。

默认规则：

- 页面标题、模块标题和下一步标题优先使用“动词 + 对象”，例如“复查结果”“查看运行状态”“重新生成笔记”“导出飞书”。
- 避免把行动标题写成状态判断句，例如“结果可以复查”“任务正在运行”“还没有可解释的任务”。
- 状态词可以出现在状态徽标、进度标签和辅助说明里，例如“已完成”“运行中”“失败”。
- 下一步提示应直接告诉用户要做什么。优先写“删除这条失败记录，然后重新提交”，不要写“可以删除这条失败记录”。
- 如果一句话不能帮助用户决定下一步动作，就不要把它放在标题或主操作区域。
- 处理详情和自动化页面尤其要用动作语言串起任务：接收材料、准备输入、生成笔记、保存产物、复查结果。
- 内部产品原则不能原样进入 UI。比如“不要做多 Agent 表演”“不是概念包装”“不是假的”应转译成用户能用的信息，例如“按执行顺序展示处理步骤”“查看真实工具调用记录”“复查失败原因和下一步动作”。
- 避免防御性文案和自证清白式表达。不要在界面里写“这不是……而是……”“不是为了……”“不是表演”。用户不需要看到产品在回应内部质疑。

## Color Rules

- 不用纯黑 `#000000` 作为页面背景。
- 不用纯白 `#ffffff` 作为暗黑模式文字大面积输出。
- 主操作只使用主品牌色或 `primary` 语义色，不要在不同页面随意更换 CTA 颜色。
- AI 摘要、模型生成、智能处理这类能力可以使用次级智能色，但不要泛化成全站装饰色。
- 不使用大面积蓝紫渐变、发光边框、霓虹阴影作为默认暗黑风格。
- 状态色必须有语义：成功、警告、错误、进行中、禁用。不要为了“丰富”随意加色。

## Component Rules

### Buttons

- 主按钮：用于提交、上传、保存、开始处理、下载等核心动作。
- 次按钮：用于导入、查看、重试、复制等辅助动作。
- 图标按钮必须有可访问标签或悬浮提示。
- 所有按钮都要有 `hover`、`focus-visible`、`active`、`disabled` 状态。
- 不要让按钮文字换行导致按钮高度跳动。

### Status Chips

- 状态标签用于辅助用户理解对象状态，不是标题本身。放在页面标题、模块标题或列表标题旁边时，字号、图标和高度必须低于标题层级。
- 标题旁状态标签默认使用 `text-[10px]` 到 `text-xs`、`h-5` 到 `h-6`、紧凑内边距；不要继承标题字号、标题行高或标题字重。
- 状态标签可以和标题在同一行，但视觉权重应明显低于标题。用户第一眼应先读到对象名称，再看到保存、失败、运行中等状态。
- 如果状态文字变长，优先缩短文案或移动到辅助说明区，不要让状态标签挤压标题或操作按钮。

### Icons

- 侧栏导航、通用操作、设置菜单等产品 UI 图标优先使用 `lucide-react`，保持同一套笔触、端点、比例和视觉重心。
- 自绘 SVG 只用于品牌 logo、平台品牌图标、或 Lucide 无法表达的特殊图形；不要用临时手绘图标补通用导航。
- 同一导航组内，图标必须在外形和语义上可区分；不要让两个入口都使用“横线列表”“文档页”“进度条”这类近似轮廓。
- 历史/记录入口优先使用历史、恢复、结果、失败状态语义；结果编辑入口优先使用编辑、铅笔、文本修改语义。
- 新增导航项时先和现有侧栏图标做一次对照，发现近似图标就换掉，不要只看图标名称不同。

### Inputs

- 标签在输入框上方，不能只靠 placeholder 当标签。
- 默认使用 `bg-surface-container-low`、`ff-border-control`、`text-on-surface`。
- Focus 使用 `border-primary/60` 或轻量 ring，不要强发光。
- 说明文字只保留必要信息。普通用户不需要理解的维护者说明应隐藏到高级设置、帮助文档或管理员入口。

### Cards

- 卡片半径遵循项目现有 scale：小半径、克制，不做大圆角气泡。
- 卡片只表达真实边界：任务项、历史项、账号信息、弹窗、编辑工具。
- 避免“为了排版好看”给每个 section 套一个大卡片。

### Tables / Admin

- 管理页优先密度和扫描效率。
- 行高稳定，数字对齐，操作列固定宽度。
- 错误和权限状态用明确文本，不只靠颜色。

### Result Editor / Transcript

- 文件名、任务名、字幕标题必须允许截断，并通过 `title` 或 tooltip 暴露全名。
- 长标题不要撑开操作按钮区。
- 长文件名、任务名、邮箱、URL 和外部标题所在的 flex/grid 链路必须逐层设置宽度约束。常用组合是父网格 `min-w-0`，内容列 `min-w-0 flex-1 overflow-hidden`，文本自身 `max-w-full truncate`。
- 只给文本节点加 `truncate` 不够；如果父级 flex item 仍按内容宽度撑开，省略号不会生效，页面会横向溢出。
- 字幕正文优先阅读节奏，时间戳列宽固定，跟随播放不能因为文本换行导致定位混乱。
- 媒体复查面板不是播放器页面。它优先服务“边听/边看、边修改当前内容”；播放器、上下文列表和辅助跳转都不能压缩当前可编辑对象。
- 文本校对、音频校对和媒体复查应按素材能力切换。没有原媒体、没有时间戳或素材类型不支持时，回到可用的校对模式，不要保留空的媒体框。

## Layout Rules

- 固定格式 UI 元素要有稳定尺寸：工具栏、按钮组、时间戳列、状态徽标、历史卡片。
- 所有多列布局必须用 `minmax(0, 1fr)` 或等效约束，避免内容列被长文本撑爆。
- 全屏页面、弹窗和抽屉必须明确滚动所有权。外层负责视口高度和遮罩时，内部应使用 `flex min-h-0 flex-col`，正文区使用 `min-h-0 flex-1 overflow-y-auto`，不要依赖页面 body 滚动兜底。
- 侧栏和主内容之间保持清晰分隔，避免大面积同色块混在一起。
- 任务入口或工作台类页面应优先显示当前可行动内容，最近活动不能压过上传、导入或链接提交入口。
- 设置和长期偏好页面按用户目标分组，不按技术服务供应商暴露复杂性。
- 应用内页面首屏优先展示用户当前要处理的对象、状态、列表、编辑区域或主要表单。不要让低信息密度的欢迎区、品牌区、说明区压过实际内容。
- 移动端如果支持，需要明确单列折叠，不依赖“自然挤压”。

## Dark Mode Checklist

每次改 UI 前后检查：

- 浅色和暗黑模式都可读。
- 对核心页面、弹窗、记录列表、结果编辑页、处理详情页等真实使用场景，要看实际截图判断文字和控件对比度；不要只相信代码里的颜色类。
- 表面层级清楚，不靠纯黑背景和强阴影。
- 主要动作在暗黑模式中仍然突出。
- 输入框边界、禁用态、错误态能看出来。
- 长标题、长邮箱、长文件名不会挤压布局。
- 长标题不会造成横向滚动；需要在桌面宽屏、窄桌面和移动宽度至少各检查一次。
- Hover、focus、active、disabled 状态齐全。
- 没有新增大面积原始颜色类。
- 没有把说明性小字堆到普通用户主流程。

## Anti-Patterns

以下模式默认不要引入：

- 工具页里使用巨型海报标题。
- 应用内页面使用大面积低价值 hero：品牌 eyebrow、巨型标题、长说明文字，但没有状态、输入、列表或可执行操作。
- 暗黑模式用大片纯黑和荧光蓝紫表现“科技感”。
- 大面积渐变、发光、装饰性纹理。
- 卡片嵌套卡片。
- 每个 section 都做成浮动卡片。
- 用解释性文案替代清晰控件。
- 未先定义页面主任务，就把所有可用功能平铺到默认视图。
- 把内容、状态、操作和解释混在同一区域，导致用户分不清哪里是正文、哪里是系统说明、哪里是下一步动作。
- 让低频或高风险操作与高频主操作同级竞争，例如在结果编辑页里把“重新转录”做得和“导出”一样显眼。
- 把处理判断、模型参数、生成原因等解释信息常驻在阅读或编辑正文旁边，占用核心内容高度。
- 把处理详情页做成只有“接收材料、转录、生成、保存”的通用步骤列表，却没有当前素材为什么这么处理。
- 用模型自述或思考过程替代可复查证据卡，让用户看见很多文字但无法判断依据是否成立。
- 用状态判断句当行动标题，例如“结果可以复查”“任务正在运行”。
- 在结果编辑页默认展示大面积原媒体和完整字幕上下文列表，导致笔记正文或当前字幕编辑空间被挤压。
- 把内部设计原则直接写进 UI，例如“这不是多 Agent 表演”。
- 只优化主页面，放任登录、关于、协议、弹窗和设置维护区继续使用旧 UI 语言。
- 输入框只靠 placeholder 传达含义。
- 同一页面多套圆角、阴影、边框风格混用。
- 为了视觉丰富混用蓝、紫、绿、橙等多个强调色。
- 标题换行超过两行并挤占主要操作区。

## Validation

文档约束之外，实际改动后仍需验证：

- 前端代码变更后运行 `npm run build:frontend`。
- 如果涉及核心页面视觉变化，启动本地服务并检查至少一个桌面视口截图。
- 涉及暗黑模式、低对比文字、状态卡片、列表、表单或弹窗时，要在浅色和暗黑模式各检查一次；重点看截图里是否有“能看到但读不清”的灰字、弱边框或禁用态误判。
- 涉及长文本显示时，用长文件名、长邮箱、长任务名测试。
- 如果截图暴露了新的可读性问题，把问题类型补回本设计系统，优先记录“什么视觉现象会误导用户”，不要只记录某个具体 CSS 类。

本规范不是替代判断的模板。它的作用是防止 FluentFlow 在长期迭代中出现主题漂移、暗黑模式失控、工具页文字过大和控件风格不统一。
