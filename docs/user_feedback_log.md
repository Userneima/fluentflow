# FluentFlow 用户反馈记录

本文档用于记录真实用户试用 FluentFlow 的过程和反馈。它不是宣传材料，也不是 bug 列表；它的作用是回答长期产品验证里的几个问题：

- 谁真的用过？
- 用了什么材料？
- 是否独立跑通上传、转录、摘要和导出？
- 卡在哪里？
- 使用前后的行为有没有变化？
- 哪些反馈能转化为下一轮产品判断？

记录原则：

- 只记录用户明确表达或真实观察到的行为，不替用户脑补满意度。
- 不粘贴完整音视频、完整转录稿、完整笔记或隐私内容。
- 如果要引用用户原话，先脱敏。
- 不把“完成导出”包装成“认可质量”，除非用户明确表达。
- 不把自己的测试样本当成外部用户反馈。

## 反馈状态

| 状态 | 含义 |
| --- | --- |
| `planned` | 已计划邀请，还没使用。 |
| `observed` | 已观察到用户使用过程。 |
| `completed` | 用户完成了一个完整任务闭环。 |
| `blocked` | 用户因产品、配置或理解问题未完成。 |
| `follow_up` | 已使用，但需要后续追问或复测。 |

## 反馈记录

当前还没有外部用户反馈记录。第一轮公开试用建议先邀请 3-5 个真实用户，每人处理 2-3 个真实材料，不要只挑短视频或容易成功的样本。

## 试用任务建议

第一轮试用不要让用户自由探索太多功能，直接给一个最小任务：

1. 打开产品并登录。
2. 上传一个真实学习/工作材料。
3. 等待转录完成。
4. 查看完整转录。
5. 查看 AI 摘要。
6. 下载或导出到飞书。
7. 回答三件事：
   - 是否愿意下次继续用？
   - 哪一步最不确定？
   - 生成笔记是否减少了整理时间？

建议样本分布：

| 类型 | 数量 | 原因 |
| --- | --- | --- |
| 5-15 分钟短视频 | 2-3 个 | 验证新用户能否快速跑通。 |
| 30-60 分钟课程/讲座 | 2-3 个 | 验证核心使用场景。 |
| 1-3 小时长材料 | 1-2 个 | 验证 ElevenLabs 云端转录、等待体验和长笔记质量。 |
| 已有字幕 SRT/TXT | 2 个 | 验证跳过 STT 的笔记链路。 |

## 反馈问题清单

### 使用前

- 你平时如何处理长视频、课堂录音或讲座录音？
- 一次完整整理大概要多久？
- 最痛苦的是转录、摘要、格式整理、归档，还是复习？

### 使用中

- 你是否知道当前系统在做什么？
- 哪一步让你觉得像卡住了？
- 你是否理解“仅转录”“高保真笔记”“导出飞书”的区别？
- 如果失败了，你是否知道下一步该怎么做？

### 使用后

- 你会保留这份笔记吗？
- 你还需要人工改哪些部分？
- 你会把它放进自己的知识库吗？
- 和原来的处理方式相比，省了哪一步？
- 你下次还会用它处理什么材料？

## 简历可用口径

可以写：

- 邀请 X 名真实用户处理 X 份真实学习/工作材料。
- 观察到 X/Y 的用户能独立完成上传、转录、摘要和导出闭环。
- 基于用户反馈定位 X 类核心卡点，例如等待状态不清晰、长笔记漏点、导出配置理解成本高。
- 根据反馈优化任务状态展示、转录编辑保存、云端长音频路径等关键体验。

必须谨慎：

- “用户满意度提升”需要明确评分或对比问卷，当前不能直接写。
- “笔记可用率”需要用户明确标记可用或不可用，当前不能从导出行为推断。
- “人工校对耗时下降”需要校对前后计时或用户明确估计，不能从编辑记录自动推出。
