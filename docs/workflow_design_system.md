# FluentFlow Workflow Design System

本文档约束 FluentFlow 的产品路径、页面职责、状态反馈和恢复动作。它不负责视觉样式，视觉层面看 `docs/ui_design_system.md`；它负责回答“用户下一步该去哪、为什么、能做什么”。

## Product Read

FluentFlow 是长任务工具。用户不是来浏览功能，而是要把视频、音频、字幕或链接变成可复查、可导出、可继续加工的笔记。

默认工作流判断：

- 核心目标：减少从素材到结果的断点。
- 核心风险：任务状态分散、入口重复、失败原因不可读、历史和当前任务口径不一致。
- 核心原则：用户永远应该知道当前任务在哪里、系统正在做什么、失败后下一步是什么。

## When To Use

改动以下内容前先看本文档：

- 任务入口页、导入页、记录列表、处理详情、结果编辑页之间的跳转。
- 最近任务、记录列表、当前任务摘要、处理详情等任务入口和状态摘要。
- 素材提交、来源解析、字幕导入、生成笔记、重试、取消、删除、导出。
- 任务状态、错误提示、进度展示、失败恢复、结果入口。
- 会改变用户对“任务在哪里、结果在哪里、下一步是什么”理解的任何改动。

## Core Rules

### 1. One Task, One Source Of Truth

同一条任务在不同页面里必须使用同一套状态语义。不要让入口页、记录列表、处理详情、结果页各自维护不同版本。

默认判断：

- 记录列表是长期任务入口。
- 处理详情是单条任务的实时解释入口。
- 结果编辑页是复查和二次加工入口。
- 入口页只展示最近任务摘要，不成为另一套历史系统。

如果某个页面需要缓存任务，只能作为快速首屏，不得覆盖后端更新后的真实状态。取消、删除、失败、完成这类用户刚触发的动作要优先保留本地结果，避免轮询把旧状态闪回。

### 2. Page Responsibility Must Stay Clear

每个页面只承担一个主要职责：

| 页面类型 | 主要职责 | 不应该承担 |
| --- | --- | --- |
| 任务入口页 | 选择素材、提交任务、查看最近摘要 | 展示复杂进度、解释完整失败原因 |
| 导入页 | 接收一种明确输入并产出任务 | 复刻其他输入类型的完整流程 |
| 记录列表 | 找到过去、失败和进行中的任务 | 展开完整判断链 |
| 处理详情 | 解释任务阶段、失败原因、下一步 | 编辑笔记正文 |
| 结果编辑页 | 复查转录、修改字幕、阅读和导出笔记 | 展示处理队列、运行日志或系统排障信息 |
| 设置页 | 长期偏好、凭证、本机配置 | 单次任务判断或失败恢复 |

当一个页面开始承担两个以上主要职责时，优先拆入口、改跳转，或者把次要信息折叠。

### 3. Entry Points Should Not Multiply

同一个目标不要在同一屏放多个强入口。重复入口会让用户怀疑它们是不是不同功能。

默认规则：

- “查看结果”只保留一个主入口。
- 记录列表入口通常只保留页面左上角或导航入口，不在任务卡里重复放大按钮。
- 处理详情作为解释入口，不和“编辑结果”抢主按钮。
- 完成态优先展示结果入口；失败态优先展示恢复入口；运行态优先展示取消或查看详情。

### 4. Progress Must Be Explainable

长任务不能只显示百分比。用户更关心卡在哪一步、为什么慢、能不能离开页面。

任务进度至少要能表达：

- 当前阶段：解析、获取、音频、转录、笔记、导出。
- 当前状态：排队中、运行中、已完成、失败、已取消。
- 是否后台继续：离开页面后是否还会继续处理。
- 如果停住：是等待队列、平台下载慢、转录中、AI 服务失败，还是需要用户操作。

不要展示假的精确进度。没有真实进度时，用阶段状态和已耗时解释。

### 5. Failure Must Lead To An Action

错误提示不应该只是报错文本。每个失败态都要尽量回答：

- 发生了什么？
- 这会不会损坏已完成部分？
- 用户下一步能做什么？

优先使用用户语言，不直接暴露 `HTTP 401`、traceback 或底层异常。技术细节可以放进处理详情的高级信息或日志里。

常见恢复动作：

- 补齐账号、授权或额度后重试。
- 切换可用处理路线后重试。
- 重新选择或重新提交素材。
- 基于已完成的中间结果继续生成。
- 删除不可恢复或已过期的失败记录。

### 6. Results And Artifacts Need Clear Ownership

用户需要知道“最后生成了什么、在哪打开、能否下载”。

默认结果层级：

- 结果编辑页：主阅读和复查入口。
- 产物下载：字幕、文本、笔记、媒体源等可保存文件。
- 外部同步：同步入口和同步状态，不替代本地结果。
- 处理详情：解释产物为什么存在或为什么缺失。

如果某个产物缺失，页面要说明原因，例如源文件过期、字幕未生成、AI 笔记失败、外部授权失效。

### 7. Old Workflow Residue Needs A Sweep

发现旧入口、旧文案或旧状态语义时，不只删可见按钮。需要顺手检查：

- 入口文案和导航命名。
- 相关缓存、排序和过滤逻辑。
- 空状态和失败状态。
- 处理详情、记录列表、最近任务是否还引用旧语义。
- Agent API / MCP 是否需要同步。
- 回归测试或 fixture 是否仍保留旧叫法。

关键词扫描只是线索，删除前要确认是否仍代表当前产品概念。

## Workflow Checks Before Finishing

涉及工作流的改动，完成前至少回答：

- 用户从哪里进入，完成后去哪里？
- 页面之间是否出现重复入口或重复状态源？
- 失败后是否有明确下一步？
- 运行中离开页面后，用户能否在记录列表或处理详情找回任务？
- 最近任务、记录列表、处理详情是否对同一任务显示同一状态？
- 是否需要更新 `docs/agent_mcp_parity.md` 涉及的 Agent API / MCP 能力？
- 是否需要在 `docs/changelog.md` 记录用户可见或维护者可见变化？
